# AI-SDLC 发布会单屏 Deck 设计方案

  - 键盘左右翻页的单屏演示形态

## 2. 背景与问题定义

当前 `launch-web` 已具备完整内容区块与视觉基线，但呈现形式仍接近长滚动网页。对发布会场景而言，这会带来三类问题：

1. 叙事节奏由滚动控制，不像 PPT 式逐页推进，现场讲述难以控节奏。
2. 信息分布过散，尤其在超宽屏下会稀释重点，观众难以快速抓住结论。
3. 内容表达偏网页说明文，英文标签较多，不够贴近中文技术受众的阅读习惯。

本次设计将 `launch-web` 从“可滚动落地页”重构为“单屏可翻页 deck”，并以中文技术演讲的表达方式重新组织内容。

## 3. 目标与非目标

### 3.1 目标

本方案必须同时满足以下目标：

1. 交互形态从长滚动网页切换为单屏演示 deck。
2. 用户可通过 `ArrowLeft / ArrowRight` 进行主翻页，并支持 `PageUp / PageDown / Space / Home / End` 辅助导航。
3. 页面默认锁定在单屏舞台中，不出现纵向阅读作为主交互。
4. 内容组织更适合中文技术人员到技术管理者的现场阅读。
5. 页面讲清楚 AI-SDLC 的工程对象、问题定义、架构区别、核心价值，不落入泛化概念或咨询话术。
6. 架构页必须正面回答“这不就是 workflow / agent 编排吗”的质疑。

### 3.2 非目标

本次设计不包含以下内容：

1. 不引入 `project/meeting` 的 presenter / audience 双页控制。
2. 不新增后台数据、远程配置或内容管理系统。
3. 不把页面做成逐条点击 reveal 的复杂演讲器。
4. 不扩展为多语言版本或多套主题皮肤。

## 4. 核心设计结论

### 4.1 形态结论

`launch-web` 应采用单屏 stage 形态，类似演示 deck，而非滚动网页。

- 每页对应一张 slide。
- 页切换采用横向 motion，不使用纵向滚动过渡。
- 顶部原有锚点导航改为 deck chrome，承担品牌、章节、页码和进度呈现。
- URL 使用 `?slide=n` 记录当前页，便于刷新后保持定位。

### 4.2 视觉方向结论

视觉方向采用“浅底舞台式 + Spine Led”结构：

- 保留当前页面的浅色、玻璃感和冷静技术气质。
- 增加舞台感 frame、稳定页码和章节 spine，使其明显区别于普通网页。
- 不转向 `project/meeting` 的深色演示风格。

### 4.3 内容方向结论

内容表达优先面向中文技术受众：

- 先用中文把问题说清楚，再用最少量英文名词做技术锚点。
- 每页以“标题 + 主句 + 2-5 条支撑信息”为基本单位。
- 避免“大词正确但缺少抓手”的表达，例如“能力底座”“演进闭环”单独出现时不承担解释责任。
- 所有价值点需落到工程结果，例如假完成、人工补洞、失败回写、证据留存。

## 5. 交互与运行模型

### 5.1 翻页规则

- 下一页：`ArrowRight`、`PageDown`、`Space`
- 上一页：`ArrowLeft`、`PageUp`
- 第一页：`Home`
- 最后一页：`End`
- 当用户已在第一页继续触发“上一页”操作，或已在最后一页继续触发“下一页”操作时，行为为 no-op，不循环跳转。

### 5.2 页面行为

- 视口固定为 `100vh`，默认不出现纵向滚动作为阅读方式。
- 当前页状态与 URL 同步，格式为 `?slide=<index>`。
- `slide` 采用 1-based 编号：
  - `?slide=1` 表示第一页
  - 缺省参数时默认进入第一页
  - 非数字、零、负数或超出页数范围时回退到第一页
- 桌面端键盘事件是主控制方式，不向用户展示显式 `上一页 / 下一页` 翻页控件。
- 桌面端可保留页码与进度展示，但这些信息默认不承担可见翻页按钮职责。

### 5.3 响应式约束

- 桌面与超宽屏下维持稳定舞台比例。
- 窄屏下仍保持一页一屏，但 spine 可压缩为顶部信息条。
- 响应式交互切换以视口宽度为唯一判断条件：
  - 视口宽度 `> 900px` 时按桌面规则处理，隐藏显式翻页控件
  - 视口宽度 `<= 900px` 时进入窄屏模式，显示显式 `上一页 / 下一页` 翻页控件
- 以上分支在窗口 resize 与设备方向切换后需要重新计算并即时生效。
- 窄屏模式不退化成长滚动页面；如空间不足，可降低装饰密度，但保留 slide 概念。
- 窄屏模式必须提供至少一种无键盘翻页方式，默认要求显式 `上一页 / 下一页` 点击控件；如后续实现 swipe，也只作为增强，不替代可见控件。

## 6. 布局系统

### 6.1 Spine Led 舞台结构

页面采用左右分区：

1. 左侧为稳定的 spine：
   - 品牌标识
   - 当前大章节
   - 当前页标题
   - 页码与总页数
   - 进度指示
2. 右侧为主舞台：
   - 当页主信息块
   - 关联图示或对照结构
   - 限量辅助说明

spine 的职责是维持观众的方位感，而不是承担多入口导航。

### 6.2 信息密度约束

每页需满足以下原则：

- 仅有一个主信息中心。
- 正文优先使用短句而非长段。
- 单页正文控制在适合投屏快速扫读的密度内，目标感知为约 70-110 个汉字。
- 英文仅作为术语补充，不主导阅读路径。

### 6.3 动效约束

- 翻页时旧页轻退、新页推进，重点是方向感，不是炫技。
- 单页内部仅允许轻量入场顺序感，不使用复杂 click reveal。
- 动效必须服务“像 deck 一样切页”，不能重新引入网页滚动感。

## 7. 内容结构

Deck 固定为 8 页，结构如下。

### 7.1 Slide 1

- 标题：`AI-SDLC 是把任务、产物、校验、回退绑在一起的交付框架`
- 主句：`我们要改的不是聊天框，而是交付链路本身。`
- 信息点：
  - 任务、产物、校验必须绑定
  - 人保留最终决策权
  - 先定义交付对象，再讨论 AI 能力

### 7.2 Slide 2

- 标题：`模型够用了，交付还在人工兜底`
- 主句：`今天卡住效率的，不是生成能力，而是任务分解、上下文续接和完成确认。`
- 信息点：
  - 会写不等于会交付
  - 跨步骤容易丢上下文
  - 很多“完成”仍要人补洞

### 7.3 Slide 3

- 标题：`问题不在模型聪不聪明，而在偏差有没有被关住`
- 主句：`一旦没有状态和校验，AI 的不确定性就会变成“看起来完成”。`
- 信息点：
  - 规则悬空会导致输出漂移
  - 结果不可追踪，交接易断裂
  - 错误最终仍回到人工兜底

### 7.4 Slide 4

- 标题：`AI-SDLC 的三件事：阶段、状态、校验`
- 主句：`它不是把任务排顺序，而是让每一步都有输入、产物、判定和回退。`
- 信息点：
  - 阶段负责推进
  - 状态负责记忆和续接
  - 校验负责决定继续还是停下

### 7.5 Slide 5

- 标题：`这不是普通工作流编排`
- 主句：`普通 workflow 只管下一步，AI-SDLC 还要管完成定义、失败处理和默认约束。`
- 信息点：
  - 状态不是日志，而是可继续执行的上下文
  - 校验不是审计，而是继续推进的门槛
  - 失败不是结束，而是带回退规则的回路入口

### 7.6 Slide 6

- 标题：`执行完成不算完成，验证通过才算`
- 主句：`每个任务都必须留下可检查的证据，没有通过验证就不能算交付。`
- 信息点：
  - 产物必须可检查
  - 失败必须回退
  - “差不多”不能被包装成完成

### 7.7 Slide 7

- 标题：`失败不只是修补，要回写成规则`
- 主句：`一次失败只有进入归因、更新规则和工具之后，才算真正被系统吸收。`
- 信息点：
  - 区分规则、工具、流程三类问题
  - 修正结果必须写回默认路径
  - 下次执行默认继承修正结果

### 7.8 Slide 8

- 标题：`把 AI 从助手变成研发底座`
- 主句：`真正的差别不是有没有 AI，而是结果能不能被验证、被继承、被持续复用。`
- 信息点：
  - 完成定义更清楚
  - 质量控制更前置
  - 经验可沉淀为系统能力

## 9. 验收口径

实现完成后，至少需要满足以下验收条件：

1. 页面进入后默认呈现单屏 slide，而不是长滚动网页。
2. `ArrowLeft / ArrowRight` 翻页行为稳定，页码与进度同步变化。
3. `?slide=n` 能正确恢复到指定页。
4. `?slide` 非法值会稳定回退到第一页。
5. 页面在桌面宽屏下具有明显的舞台式 deck 结构。
6. 桌面端默认不展示显式 `上一页 / 下一页` 翻页控件。
7. 视口宽度 `<= 900px` 时，用户仍可通过可见控件完成翻页。
8. 8 页内容均符合本方案定义的中文标题和主句方向，不退回英文导航页风格。
9. 架构页能够清晰体现其与普通 workflow 编排的区别。
10. 测试与必要的浏览器验证通过。

## 10. 风险与实现提醒

### 10.1 风险

1. 如果沿用当前 section DOM 直接强行套翻页逻辑，容易保留滚动网页遗留结构，导致交互和布局互相打架。
2. 如果文案再次回到抽象大词，最终成品会像概念展示页，而不是能支撑现场讲述的技术 deck。
3. 如果舞台装饰过重，窄屏下会压缩可读性。

### 10.2 实现提醒

1. 应优先按“slide 容器 + slide 数据 + 当前页状态”重建骨架，而不是继续围绕锚点 section 修补。
2. 需要把 `project/meeting` 的翻页模型作为交互参考，但只复用单 deck 模式，不引入双页结构。
3. 验证逻辑需兼顾键盘交互、URL 恢复与典型桌面视口截图。
