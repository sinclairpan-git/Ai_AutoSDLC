# 框架自迭代开发与发布约定

> 适用范围：在 AI-SDLC 框架仓库内，直接开发框架自身，并使用框架自己的 CLI、trace、约束和 backlog 持续优化框架本身。

## 目标

这份文档用于澄清一个经常混淆的问题：

- 我们是在 **AI-SDLC 框架仓库内部** 开发 AI-SDLC 本身。
- 我们希望用 Harness Engineering 的思路，把框架的日常工作过程变成可观察、可追溯、可归因、可改进的闭环。
- 我们需要明确：开发、提交、push、PR、merge、pull、release 各自改变的到底是什么，以及怎样保证项目内正在执行的 `ai-sdlc` 始终对应当前最新源码。

## 核心原则

### 1. 开发态 CLI 与发布态 CLI 必须分开理解

- **开发态 CLI**：当前仓库源码直接运行出来的 CLI。
- **发布态 CLI**：通过 tag / release / wheel / zip 安装给外部项目使用的 CLI。

在框架仓库里做自迭代开发时，默认讨论的应当是 **开发态 CLI**，不是发布态 CLI。

### 2. 项目内开发时，源码仓库就是 CLI 的真身

在本仓库里，最稳的执行方式是：

```bash
uv run ai-sdlc ...
```

这意味着：

- 你运行的 CLI 直接来自当前工作区的 [pyproject.toml](../pyproject.toml) 和 [src/ai_sdlc](../src/ai_sdlc)。
- 只要你继续在当前仓库里运行 `uv run ai-sdlc ...`，你执行的就是当前工作区最新代码。
- `git push` 不会让本地 CLI 自动变新，也不会让它变旧。

### 3. 不要把“push”误当成“同步本地 CLI”

- `commit`：把当前本地修改固化成本地提交。
- `push`：把你已经拥有的本地提交上传到远端。
- `pull`：把远端有、但本地还没有的提交拉回本地。

所以：

- **本地改完代码并提交后**，项目内 CLI 已经是最新，不需要 pull。
- **刚 push 完分支后**，项目内 CLI 仍然是你本地刚写的版本，不需要 pull。
- **只有远端出现了你本地没有的新提交**，才需要 pull 或等价同步动作。

### 4. GitHub / PR / Actions 只属于本仓库工程层，不属于框架通用契约

- 本仓库当前使用 GitHub、Pull Request、GitHub Actions 与 Codex review 作为自迭代工程实现；这些动作只约束 **AI-SDLC 框架仓库自身** 的协作与发布流程。
- 框架对外分发的通用约束应保持平台无关，只描述“代码审查”“远端集成门禁”“回归验证”“主线合入前校验”等抽象能力，不把 GitHub 专有对象或命令写成必选前提。
- 因此，`src/ai_sdlc/rules/*`、`templates/*`、`presets/*` 不得把 GitHub、`gh pr`、GitHub Actions、`@codex review`、GitHub merge 语义写成所有项目都必须遵守的默认规则。
- GitHub 相关内容只能保留在本仓库的工程层配置与自迭代文档，例如 `.github/workflows/*`、本文件，以及只服务于本仓库协作方式的操作说明。
- 若未来消费仓库使用公司内代码仓、内部 review 系统或非 GitHub CI，只要满足等价的审查、验证与合入治理结果，即符合框架要求；不需要为了适配框架而引入 GitHub 依赖。

## Harness 自迭代闭环

我们采用的不是“凭印象优化框架”，而是基于运行证据的闭环优化。

### 1. 观察日常工作过程

- raw trace 位于 `.ai-sdlc/local/telemetry/`
- 运维阅读面位于 `.ai-sdlc/project/reports/telemetry/`
- 只读诊断面通过 `status --json`、`doctor`、`workitem branch-check` 与 `workitem truth-check` 暴露

目标不是只看“有没有报错”，而是回到实际执行轨迹，回答：

- 跳过了哪些命令
- 执行了哪些动作
- 漏掉了哪些动作
- 哪些结论来自真实 evidence，哪些只是 inferred

### 2. 发现违约、缺陷和治理缺口

除了 trace，还要结合：

- `uv run ai-sdlc verify constraints`
- `uv run ai-sdlc status --json`
- `uv run ai-sdlc doctor`
- `uv run ai-sdlc workitem truth-check --wi specs/<WI> --rev <branch-or-commit>`
- `uv run ai-sdlc workitem branch-check --wi specs/<WI>`
- code review / gate / close-check / 回归测试
- 人工补记的 telemetry evidence

一旦发现框架级缺口，要继续追问：

- 这是 prompt / context 问题，还是 rule / policy 问题
- 是 workflow 缺口，还是 tool / middleware 缺口
- 根因是什么
- 怎样才能防止同类问题再次出现

### 3. 将现象沉淀成长期优化输入

当问题暴露出框架层缺口时，必须把它沉淀到 [framework-defect-backlog.zh-CN.md](./framework-defect-backlog.zh-CN.md)，而不是只留在会话备注里。

一旦已经明确说出“这是违约 / 这是框架缺口”，下一动作默认应当是：

1. 写 backlog 条目
2. 跑必要的只读校验
3. 再讨论补正路径、修复方案或实施计划

不得先继续解释、继续实施，或等待用户二次提醒后才补录 backlog。

最小闭环应当是：

1. 观察现象
2. 交叉印证 trace / constraints / evidence / backlog
3. 定位根因
4. 设计防复发机制
5. 实现修复
6. 增加回归验证
7. 把规则、文档、工具和测试更新到位

### 4. 新 framework capability 的设计输入不等于法定执行入口

- 新 framework capability 的默认 canonical 入口应直接位于 `specs/<WI>/`；在仓库已完成 `uv run ai-sdlc init .` 的 formal bootstrap 前提下，优先使用 `uv run ai-sdlc workitem init --title "<标题>"` 或等价 direct-formal 入口，一次性生成 `spec.md`、`plan.md`、`tasks.md`。
- `docs/superpowers/specs/*.md` 与 `docs/superpowers/plans/*.md` 负责 architecture baseline 与 planning frame，它们是设计输入，不是法定执行真值，也不是必须先写的一套 canonical 文档。
- 任何“可以直接开做”“从 Task 1 开始实施”之类的表述之前，必须先把能力收敛到正式 work item：`specs/<WI>/spec.md`、`specs/<WI>/plan.md`、`specs/<WI>/tasks.md`。
- 进入 execute/close 后，法定执行真值应当回到 work item 目录内的 `tasks.md`、`task-execution-log.md` 与主线提交链；不得继续把 `docs/superpowers/plans/*.md` 或聊天结论当作实施完成依据。
- 宿主 skills（例如 `brainstorming`、`writing-plans`）给出的 terminal state、implementation handoff、执行选项提示，只是宿主工作流提示，不是仓库 execute 授权；它们不能覆盖本仓库的阶段真值。
- 当用户仍在 review spec/plan、要求继续修正文档、或追问流程约束时，默认动作必须继续停在 docs/rules/review 层，而不是把“下一步”自然推进成编码。
- 只有在 **用户明确要求进入实现**，且仓库内 `specs/<WI>/tasks.md`、门禁与相关前置都已满足时，才允许从 plan freeze / plan review 进入开发编码；否则应明确说明当前仍处于 review / 对账 / docs-only 状态。

## 日常开发流程

### 1. 每一轮开发从最新 `main` 开始

在开始新一轮框架开发前：

```bash
git switch main
git pull --ff-only origin main
```

如果你使用 worktree，也应当从最新 `main` 派生新的分支或 worktree，而不是复用旧的脏基线。

### 2. 从最新基线创建分支或 worktree

常见做法：

```bash
git switch -c feature/<topic>
```

或：

```bash
git worktree add .worktrees/<topic> -b feature/<topic> main
```

补充约束：

- `codex/*`、临时 topic 分支和 worktree 只代表隔离执行容器，不代表主线真值。
- 如果某条能力只在 scratch/worktree 分支存在，实现状态必须明确表述为“分支已实现”，不得外推成“主线已兑现”。
- close 前必须说明该分支/worktree 的 disposition：`merged`、`archived` 或 `deleted`。

### 3. 在当前工作区内运行开发态 CLI

进入该分支或 worktree 后，统一使用：

```bash
uv run ai-sdlc ...
```

这时：

- 该 worktree 里的源码就是 CLI 的来源
- 该 worktree 里的修改会直接影响 `uv run ai-sdlc ...` 的行为
- 不需要额外“从远端再拉一遍 CLI”

### 4. 开发过程中持续做自举验证

至少应包含：

```bash
uv run ai-sdlc workitem init --title "<新的 framework capability 标题>"
uv run ai-sdlc verify constraints
uv run ai-sdlc status --json
uv run ai-sdlc doctor
uv run ai-sdlc workitem truth-check --wi specs/<WI> --rev <branch-or-commit>
uv run ai-sdlc workitem branch-check --wi specs/<WI>
uv run pytest -q
```

如果改动了依赖、打包配置或虚拟环境状态，再执行：

```bash
uv sync
```

说明：

- `workitem init` 的前提是当前仓库已完成 formal bootstrap；若只有历史 `.ai-sdlc` 痕迹但缺 `.ai-sdlc/project/config/project-state.yaml`，先执行 `uv run ai-sdlc init .`。
- `workitem init` 只在创建新的 formal work item 时使用一次；它的目的不是开始 execute，而是把 canonical spec/plan/tasks 直接落到 `specs/<WI>/`。
- 如果已经存在 formal work item，就继续围绕现有 `specs/<WI>/` 文档工作，不要再补写第二套 `docs/superpowers/*` canonical 文档。

### 5. 发现框架缺口就登记 backlog

如果在开发过程中发现：

- 规则被跳过
- 命令执行顺序错乱
- 状态判断与真实交付不一致
- 只读命令发生了副作用
- trace / evidence / status 面出现不一致

则应把现象、影响、根因和未来杜绝方案写入：

- [framework-defect-backlog.zh-CN.md](./framework-defect-backlog.zh-CN.md)

必要时再补：

- 设计说明
- 缺陷分析文档
- 回归测试

### 6. 提交、push、PR、merge

在当前分支完成验证后：

```bash
git add <files>
git status --short
git diff --cached --stat
git commit -m "<message>"
git push origin <branch>
```

此时：

- 本地工作区里的开发态 CLI 仍然是最新
- **不需要因为刚 push 了，就再 pull 一次**
- Git 写步骤保持串行：`git add -> git status/diff -> git commit -> git push`
- 不要把这些命令交给并行工具同时发起；若遇到 `.git/index.lock`，先确认是否仍有活跃 Git 进程，禁止默认删锁

开 PR、修 review、补提交后继续重复上述流程即可。

### 6.1 带 `python-runtime/` 的离线包必须按发布清单收口

当一次 release / PR 变更触及以下任一项时：

- `packaging/offline/build_offline_bundle.sh`
- `packaging/offline/install_offline.*`
- `packaging/install_online.*`
- `packaging/offline/README.md`
- 离线 bundle 的 `python-runtime/` 分发口径

则不能只写“离线包已更新”或“README 已同步”，还必须按下列清单收口：

- [packaging/offline/RELEASE_CHECKLIST.md](/Users/sinclairpan/project/Ai_AutoSDLC/packaging/offline/RELEASE_CHECKLIST.md)

尤其是当对外口径包含“目标机无需预装 Python 3.11+”时，必须显式确认：

1. bundle 内已真实包含匹配平台的 `python-runtime/`
2. `bundle-manifest.json` 已标记 `python_runtime_bundled`
3. 至少有一条“无系统 Python”安装冒烟证据
4. Release Notes / README / offline README 对该口径一致

缺任一项，都不得把该能力表述为已发布完成。

### 6.2 分支已实现不等于主线已兑现

这是框架自迭代里最容易混淆的真值边界：

- 某个 `feature/*`、`codex/*` 或 worktree 分支上已经有代码、测试、甚至局部文档，**不等于** `main` 已经兑现该能力。
- 只有当实现进入 `main`，并与 work item 的 `tasks.md`、`task-execution-log.md`、close-out 结论对齐后，才能把它表述为主线已完成。
- 如果某分支决定不合 `main`，也必须显式写成 `archived` 或 `deleted` disposition，而不是让它长期处于“存在但没人负责解释”的状态。

推荐的最小收尾顺序是：

1. 完成当前批次实现与验证
2. 若当前在核对非当前 checkout 的 branch/commit，先执行 `uv run ai-sdlc workitem truth-check --wi specs/<WI> --rev <branch-or-commit>`，确认目标 revision 的阶段真值
3. 再执行 `uv run ai-sdlc workitem branch-check --wi specs/<WI>`，确认当前 WI 关联 branch/worktree 的未处置项
4. 决定当前关联 branch/worktree 的 disposition（`merged / archived / deleted`）
5. 更新 `task-execution-log.md`、`tasks.md` 与 close-out 说明
6. 再执行 `close-check` / `verify constraints`
7. 最后再对外表述“已收口”或“仍为分支态”

### 7. PR 合并后，再同步本地 `main`

当 PR 通过 GitHub 被 merge 后，远端 `main` 往往会产生一个你本地没有的新提交，尤其是 `squash merge` / `rebase merge`。

这时才需要同步：

```bash
git switch main
git pull --ff-only origin main
```

同步的目的不是“让当前分支里的 CLI 变新”，而是：

- 让下一轮开发的基线回到最新主线
- 避免下一轮 feature 基于过时 `main`

## push / merge / pull / release 的精确关系

| 操作 | 改变了什么 | 项目内 CLI 要不要再 pull |
| --- | --- | --- |
| 本地修改源码 | 当前工作区文件变了 | 不需要 |
| 本地 commit | 本地历史变了 | 不需要 |
| push 分支 | 远端分支多了你已有的提交 | 不需要 |
| PR review 后继续本地提交 | 当前工作区和本地分支继续变化 | 不需要 |
| GitHub 上 merge PR | 远端 `main` 产生了你本地未必有的新提交 | **需要在回到 `main` 后同步** |
| 打 tag / 发 release | 发布态 CLI 更新了 | 对项目内开发态 CLI 不构成自动更新 |

## release 文档一致性收口

release 不只是一条 tag 或一份 GitHub release 正文。只要发布态 CLI、安装包、离线 bundle 或版本号发生变化，就必须做一轮入口文档一致性 sweep，至少覆盖以下文件：

- `README.md`
- `docs/releases/v0.7.0.md`（当前 staged release；后续版本升级时同步替换）
- `USER_GUIDE.zh-CN.md`
- `packaging/offline/README.md`
- `docs/pull-request-checklist.zh.md`

这轮 sweep 至少要明确两件事：

- 当前发布版本是什么，以及用户应该先看哪一份 release notes
- 不同平台应该下载哪一种资产，例如 Windows 使用 `.zip`，macOS / Linux 使用 `.tar.gz`

当前 `v0.7.0` 的固定入口口径应保持一致：

- `README.md` 指向 `docs/releases/v0.7.0.md`
- Windows 使用 `ai-sdlc-offline-0.7.0.zip`
- macOS / Linux 使用 `ai-sdlc-offline-0.7.0.tar.gz`
- `packaging/offline/README.md` 与 `docs/pull-request-checklist.zh.md` 必须同步上述口径

如果 release notes 已更新，但入口文档没有同步版本入口、资产分工或安装指引，这次 release 不能算文档收口完成。

## 如何保证项目内 CLI 始终是“最新”

结论很简单：

### 场景 1：你还在当前开发分支 / 当前 worktree 里继续改

做法：

- 继续用 `uv run ai-sdlc ...`

结论：

- 这就是当前最新本地源码，不需要 pull。

### 场景 2：你刚刚 commit 或 push

做法：

- 继续在当前工作区运行 `uv run ai-sdlc ...`

结论：

- CLI 仍然就是你刚提交 / 刚 push 的代码，不需要 pull。

### 场景 3：PR 已在 GitHub 合并，你准备开始下一轮

做法：

```bash
git switch main
git pull --ff-only origin main
```

结论：

- 先把本地 `main` 更新到远端最新，再从它开下一轮分支。

### 场景 4：你改了依赖或环境

做法：

```bash
uv sync
```

结论：

- 这是同步开发环境，不是同步 Git 代码。

### 场景 5：你误用了全局安装的 `ai-sdlc`

做法：

- 停止使用全局命令
- 回到仓库根目录执行 `uv run ai-sdlc ...`

结论：

- 框架自迭代开发应避免依赖 PATH 上的旧副本。

## 常见误区

### 误区 1：我刚 push 完，所以需要 pull 一次，才能让项目内 CLI 变成最新

错误。

原因：

- 你 push 上去的提交，本来就是你本地已经有的提交。
- 项目内开发态 CLI 本来就来自你当前工作区源码。

### 误区 2：每次 commit 后都要重新安装一次 CLI

错误。

原因：

- 在框架仓库内，`uv run ai-sdlc ...` 直接使用当前源码。
- 只有依赖变化时，才需要 `uv sync`。

### 误区 3：release 发了，所以本地仓库里的 CLI 也自动更新了

错误。

原因：

- release 更新的是发布态安装来源。
- 你的本地仓库是否更新，取决于 Git 是否同步到了对应提交。

## 一句话原则

在 AI-SDLC 框架仓库内自迭代开发时：

- **用 `uv run ai-sdlc ...` 运行当前源码**
- **push 不等于 pull**
- **merge 之后再同步 `main`**
- **trace、违规、backlog、修复、回归测试必须形成闭环**
