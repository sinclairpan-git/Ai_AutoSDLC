## 合议结论

1. V0.7.0 Windows 普通用户路径未完成“空项目到前端闭环”。两条场景均只执行成功 `managed-target-prepare`，失败在 `dependency-install`，`delivery_complete=false`，browser gate 无真实截图、无 baseline promotion、无二次 probe pass。
2. P0 根因不是 Node/npm 不存在，而是 Windows 下 Python `subprocess.run(["npm", ...])` 不解析 `npm.cmd`。现场 `where npm` 能找到 `C:\Program Files\nodejs\npm` 和 `npm.cmd`；本机最小复现中 `subprocess.run(["npm","--version"])` 返回 `[WinError 2]`，`subprocess.run(["npm.cmd","--version"])` 成功返回 `11.11.0`。
3. `workitem init` 已保存用户输入的“订单运营管理台”业务需求，但 `spec.md/plan.md/tasks.md` 的主语仍是 framework direct-formal 自身任务。这会让后续前端托管交付建立在错误业务语义上，应作为闭环级阻断处理，而不是普通模板体验问题。
4. adapter 问题应精确定义为“普通 PowerShell / dry-run / mutate gate 对 adapter 状态的口径不一致”。E2E 过程中曾通过手工环境信号补救到 `adapter_ingress_state=verified_loaded`，但当前项目配置又回到 `detected_ide=generic`、`adapter_activation_state=installed`、`adapter_support_tier=soft_installed`，且缺少 canonical consumption 证据；不能把某个字段短暂出现 `verified_loaded` 写成完整治理消费证明。
5. 离线安装中文乱码不是单纯 PowerShell 显示问题。仓库源文件中文正常，但解压后的 release asset 脚本文本本身已出现 mojibake，例如 `褰撳墠鐘舵€?`、`涓嬩竴姝ュ懡浠?`、`鍛戒护浣滅敤`，说明打包/物化链路存在内容漂移。
6. Release Notes、README、USER_GUIDE 对 `0.7.0` 的“empty-project frontend bootstrap release-grade”“真实依赖安装验证”“默认写出最小受控前端产物”等表述与本次 Windows E2E 事实冲突。缺陷处理必须同时修复实现和发布口径。
7. “离线包”当前只覆盖 Python/CLI 安装，不覆盖前端依赖闭环。公共路径仍需要公共 npm registry，企业路径仍需要内网 registry，且没有 `offline_strict / enterprise_registry / public_registry` 的模式声明与验收证据。
8. P0 失败 artifact 对 AI agent 不可执行：`blocked_action_ids: []`、`plain_language_blockers: []`、`recommended_next_steps: []`，没有 resolved executable、cwd、PATH/PATHEXT、stdout/stderr、异常类型、重试次数和恢复命令。
9. 遥测证据链与实际失败不一致：`latest-artifacts.json` 为空，`open-violations.json` 为空，现有 evidence 主要是 `verify-constraints` passed 记录，无法表达 managed delivery/browser gate/final proof 未闭环。
10. 修复优先级应按闭环阻断排序：先修 Windows 包管理器解析与依赖安装证据，再修前端依赖模式和业务需求分解语义，再修 adapter 状态机、失败 artifact、遥测证据、视觉验收、close/final proof 的误导性成功信号。

## 场景结果概览

- 离线包下载、校验、解压、离线安装：流程成功，但 release asset 内 `install_offline.ps1` 中文提示已损坏。
- `ai-sdlc init .`：两个空目录均成功初始化；adapter 状态在普通 PowerShell、手工环境信号、dry-run 之间存在口径漂移，不能稳定证明 canonical adapter 已被宿主消费。
- 需求录入：中文标题直接录入失败；显式 `--wi-id 001-order-ops-console` 后成功。
- 任务分解：业务输入被写入 `spec.md` 的 `输入` 字段，但 `spec.md/plan.md/tasks.md` 主体仍是 direct-formal 框架自开发模板。
- solution confirmation：企业和公共两条路径均能写入 `.ai-sdlc\memory\frontend-solution-confirmation\latest.yaml`，provider/style 选择结果符合输入。
- managed delivery apply：企业和公共两条路径均失败在 `dependency-install`，状态为 `manual_recovery_required`。
- 前端代码生成：仅生成 `managed\frontend\package.json` 和空 `src` 目录，未生成 `index.html`、`src\App.vue`、`src\generated\frontend-delivery-context.ts` 等可运行前端代码。
- 视觉验收：browser gate 正确 fail-closed，但无法执行真实视觉验收。
- 最终收口：close-check 正确返回 blocker，但 Git 前置条件、默认分支假设和 final-proof 输出语义均不适合普通空项目路径。

## 待修复问题

### E2E-WIN-001：命令行网络工具不自动继承 Windows 用户代理

严重级别：P2
影响阶段：下载 Release / GitHub API / Git fetch

现象：Windows 用户代理为 `127.0.0.1:10808`，浏览器链路可访问 GitHub，但 `git clone`、`curl https://api.github.com/...` 直连超时。显式配置 `git config --global http.proxy http://127.0.0.1:10808`、`curl --proxy http://127.0.0.1:10808` 后恢复。

根因归纳：普通 Windows 用户常把“浏览器能访问 GitHub”等同于“终端工具可访问 GitHub”。当前安装/下载文档没有把 Git、curl、PowerShell、pip 的代理继承差异作为 preflight 检查。

解决方案：
- 在离线包获取、在线安装和 GitHub API 访问说明中增加终端代理诊断。
- 给出 Windows PowerShell、Git、curl、pip/npm registry 的可复制代理配置与取消配置命令。
- 安装前诊断应明确区分“浏览器可达”和“命令行可达”。

验收标准：
- 在代理环境下，文档能引导用户分别验证 `git ls-remote`、`curl https://api.github.com`、`pip` 或 `npm` 的连通性。
- 代理未配置时失败信息指向终端代理，而不是泛化为“网络失败”。

### E2E-WIN-002：Windows 离线 release asset 内安装脚本中文提示已损坏

严重级别：P1
影响阶段：离线安装 / 新用户下一步引导

现象：`install_offline.ps1` 执行成功，但解压后的脚本内容已经包含 mojibake，输出中的中文标题显示为 `褰撳墠鐘舵€?`、`涓嬩竴姝ュ懡浠?`、`鍛戒护浣滅敤`、`鐩存帴璋冪敤`。仓库源文件 `packaging\offline\install_offline.ps1` 中文正常。

根因归纳：问题发生在 release asset 物化/打包链路或构建机编码处理阶段，而不只是 PowerShell 控制台显示 UTF-8 的问题。

解决方案：
- 检查 Windows offline zip 的生成链路，确保 `install_offline.ps1` 以 UTF-8 内容原样写入 zip。
- 对 release asset 增加脚本文本内容校验，校验关键中文锚点而不是只校验脚本可执行。
- 保留英文 fallback，但中文提示不能在中文用户路径中损坏。

验收标准：
- 解压 release zip 后，`Get-Content -Encoding UTF8 .\install_offline.ps1` 能看到 `当前状态`、`下一步命令`、`命令作用`、`直接调用`。
- Windows PowerShell 5、PowerShell 7 下执行脚本均不出现 mojibake。
- release checklist 包含 zip 内脚本内容编码校验。

### E2E-WIN-003：普通 PowerShell、dry-run 与 mutate gate 的 adapter 状态口径不一致

严重级别：P1
影响阶段：初始化 / managed delivery apply / AI agent 可执行性

现象：普通 PowerShell 非交互路径执行 `ai-sdlc init .` 后，adapter 容易落到 generic 或未验证状态，`adapter status` 可出现 `governance_activation_state=degraded` 或 `materialized_unverified`。随后 mutate 类命令会被 `host_ingress_below_mutate_threshold` 阻断。手工执行 `ai-sdlc adapter select --agent-target codex` 后仍可能只是 `materialized_unverified`；设置 `$env:OPENAI_CODEX='1'; $env:CODEX_CLI_READY='1'` 后重新 select 才可能达到 `verified_loaded`。对抗审查中在 E2E 项目复跑 `run --dry-run` 又输出 `IDE adapter (codex): installed 1 file(s)`，当前项目配置显示 `detected_ide=generic`、`agent_target=codex`、`adapter_activation_state=installed`、`adapter_support_tier=soft_installed`，缺少 `adapter_ingress_state` 和 canonical consumption 证据。

根因归纳：当前可验证加载依赖宿主环境信号，但普通 Windows 终端用户并不知道何时需要这些信号、不同 IDE/agent 的可机验证据是什么，以及 `run --dry-run` 成功为什么不等于治理激活。更严重的是，status 展示、project-config 字段、dry-run adapter materialization、mutate gate 使用的判定口径不完全一致。

解决方案：
- `adapter status` 和 mutate 阻断提示输出可复制的 Windows PowerShell 补救命令。
- 文档按普通 PowerShell、IDE 内置终端、Codex、Cursor、Claude Code 分别说明 verified path。
- 在 `managed-delivery-apply` 阻断时，把 `host_ingress_below_mutate_threshold` 翻译成用户可理解的下一步。
- `run --dry-run` 如果会物化 adapter 文件，必须在命名、输出和文档中明确；若承诺 dry-run 只读，则必须移除该写入。
- `adapter status` 必须同时展示 activation、ingress、canonical path、canonical consumption、evidence 和 degrade reason。

验收标准：
- 从普通 PowerShell 初始化后，用户能根据 CLI 提示完成可机验证或明确知道当前宿主不支持自动验证。
- `ai-sdlc run --dry-run` 输出继续明确：它只证明安全预演成功，不证明治理激活。
- mutate 命令在未 verified 时 fail-closed，并给出具体补救命令。
- `adapter status`、`run --dry-run`、mutate gate 和 `project-config.yaml` 对同一项目给出一致状态解释。
- 若 canonical consumption 仍为 unverified，不得仅凭 `adapter_ingress_state=verified_loaded` 对外宣称治理已完整加载。

### E2E-WIN-004：中文 work item 标题不能自动生成 slug，需求录入直接失败

严重级别：P1
影响阶段：需求录入

命令：`ai-sdlc workitem init --title "订单运营管理台" --input "..."`

现象：返回 `title does not produce a valid slug for work item id`。显式增加 `--wi-id 001-order-ops-console` 后可继续。

根因归纳：`slugify(title)` 当前对非 ASCII 标题会得到空 slug，`workitem_scaffold` 随后直接失败。对中文用户来说，这是首次业务需求录入的硬阻断。

解决方案：
- 中文标题无可用 ASCII slug 时自动降级生成 `NNN-workitem`、`NNN-feature` 或稳定 hash/序号型 id。
- 或在失败提示中直接给出可复制示例：`--wi-id 001-order-ops-console`。
- 如要支持中文转写，应采用确定性方案，并避免引入在线依赖。

验收标准：
- `workitem init --title "订单运营管理台"` 在 Windows/macOS/Linux 下都能成功生成合法 WI。
- 如果要求用户显式 `--wi-id`，错误提示必须包含合法示例和重试命令。
- 单元测试覆盖纯中文、中文加英文、emoji/符号标题、空标题。

### E2E-WIN-005：业务需求录入后生成的 spec/tasks 仍是框架 direct-formal 自开发模板

严重级别：P0
影响阶段：需求录入 / 分解 / AI-Coding 任务执行

现象：`spec.md` 顶部 `输入` 字段保留“订单运营管理台”业务需求，但主体用户故事仍是“作为框架维护者，我希望直接生成 formal work item 文档”，`tasks.md` 生成的是 `direct-formal baseline freeze`、`implementation scaffold and parser-friendly structure`、`docs alignment and focused verification` 等框架自身任务，而不是订单前后端业务开发任务。

根因归纳：普通业务项目模板和 framework self-development 模板没有隔离。CLI 表面上接受业务需求，实际生成的 canonical docs 却把后续 AI-Coding 目标导向框架自迭代。

解决方案：
- 区分 framework work item 与业务项目 work item 的模板、默认任务、验收标准和 execution log。
- 如果当前版本只能生成 formal placeholder，应在 CLI 输出中显式声明“尚未完成业务任务分解”，不能让用户误判为可执行业务 backlog。
- `workitem init --input` 应至少生成与输入相关的用户故事、FR/SC 草案、前后端任务骨架和待确认问题。

验收标准：
- 输入“订单运营管理台”后，`spec.md` 的用户故事、功能需求、成功标准与订单业务相关。
- `tasks.md` 至少包含前端列表/详情/编辑/筛选/批量操作、后端查询/详情/状态变更/导出、契约联调、视觉验收、收口证据等任务。
- 不得在普通业务项目中默认生成 `direct-formal` 框架自开发任务。

### E2E-WIN-006：program truth sync 在空项目中提示 prd_path 为空，但前置流程未引导用户创建/绑定 PRD

严重级别：P2
影响阶段：需求录入 / truth sync

现象：`program truth sync --execute --yes` 返回 ready，但警告 `prd_path is empty (recommended to set for traceability)`。

根因归纳：`program-manifest.yaml` 可由 `workitem init` 自动 bootstrap，但 PRD 绑定仍留空；CLI 没有说明这是可忽略警告、推荐项还是后续 close blocker。

解决方案：
- 在 `workitem init` 或 `truth sync` 输出中给出设置 PRD 的命令或说明。
- 将 warning 分级：advisory、recommended、close-blocking。
- 对空项目提供最小 PRD 生成/绑定路径。

验收标准：
- 用户看到 `prd_path is empty` 时能知道是否必须处理、何时处理、如何处理。
- 如果不是当前阶段 blocker，输出不得让用户误以为流程失败。

### E2E-WIN-007：Windows 下包管理器裸命令调用失败为 WinError 2

严重级别：P0
影响阶段：组件库安装 / 前端生成 / 视觉验收

现象：两条场景执行 `managed-delivery-apply --execute --yes` 后均失败在 `dependency-install`，blocker 为 `[WinError 2] 系统找不到指定的文件。`。企业场景 package manager 为 `npm`，registry 为 `http://npm.uedc.sangfor.com.cn/`。公共场景原策略为 `pnpm`，artifact 中出现 `delivery_package_manager_fallback:pnpm->npm`，随后仍以 `npm` 失败。现场 `where npm` 能找到 `C:\Program Files\nodejs\npm` 和 `npm.cmd`，`pnpm` 不存在。

根因归纳：`managed_delivery_apply.py` 中 `_default_dependency_installer` 直接调用 `subprocess.run(["npm", "install"], ...)`、`["pnpm", "add"]`、`["npx", ...]`。Windows Python `CreateProcess` 不会像 shell 一样自动解析 `.cmd` shim，导致 npm 在 PATH 中可见但无法由裸命令执行。

解决方案：
- 引入跨平台 command resolver：Windows 优先解析 `npm.cmd`、`npx.cmd`、`pnpm.cmd`、`yarn.cmd`，非 Windows 保持裸命令。
- 在执行前做 package manager preflight，区分 `command_not_found`、`registry_unreachable`、`auth_failed`、`package_not_found`、`lockfile_missing`。
- 对 fallback 行为输出结构化说明：请求 `pnpm` 但本机缺失时为什么改用 `npm`，以及锁文件/验证口径随之改变。

验收标准：
- Windows 下 `subprocess` 能成功执行 `npm.cmd install` 并生成 `package-lock.json`。
- 公共场景验证 `primevue`、`@primeuix/themes` 的 manifest、lockfile、node_modules resolution。
- 企业场景能继续走到 registry/包名/凭证层面的真实错误分类，而不是停在命令不可执行。
- Playwright runtime 安装命令同样解析 `npx.cmd` 或等价跨平台入口。

### E2E-WIN-008：dependency-install 失败后 artifact-generate 未执行，无法产出可运行前端代码

严重级别：P0
影响阶段：前端代码生成

现象：失败后 `managed\frontend` 只有 `package.json` 和空 `src` 目录，没有 `index.html`、`src\App.vue`、`src\generated\frontend-delivery-context.ts`。apply artifact 中包含这些待写 payload，但实际未落盘。

根因归纳：action plan 将 `artifact-generate` 置于 `dependency-install` 和视觉 runtime 安装之后；任一 required dependency action 失败就提前返回 `manual_recovery_required`。这与 README/USER_GUIDE 中“会写出最小受控前端产物”的用户预期冲突。

解决方案：
- 产品语义二选一并在 CLI 中明确：
  - 若依赖安装是硬前置，则结果摘要必须明确“代码未生成，当前只写了 package.json，可恢复命令为 ...”。
  - 若希望先给用户可打开骨架，则将无依赖 scaffold 先落盘，再把依赖安装和真实 provider runtime 验证作为后续 gate。
- artifact 中的未执行 payload 应标注为 `planned_not_materialized`。

验收标准：
- 依赖失败时，CLI 摘要、latest.yaml、plain language blocker 都明确列出未生成的目标文件。
- 修复后成功路径实际存在 `managed/frontend/index.html`、`src/App.vue`、`src/generated/frontend-delivery-context.ts`。
- browser gate 只在真实入口文件存在时进入执行，否则给出结构化 blocker。

### E2E-WIN-009：公共组件库路径选择成功，但真实公共安装未被验证

严重级别：P1
影响阶段：公共组件库场景

现象：solution confirmation 能正确选择 `public-primevue + modern-saas`，但依赖安装失败，未验证 `primevue` 和 `@primeuix/themes` 的真实安装、lockfile 和运行时。

根因归纳：当前 E2E 只能证明 provider/style 决策落盘，不能证明公共组件库闭环。包管理器调用失败遮蔽了 registry、lockfile、runtime import、browser rendering 等后续问题。

解决方案：
- 修复 Windows 包管理器调用后补跑公共路径。
- 将公共路径验证拆成 registry 可达、lockfile、node_modules package manifest、runtime import、browser rendering 五类证据。

验收标准：
- latest.yaml 记录公共依赖安装成功证据。
- `package-lock.json` 或选定包管理器 lockfile 存在。
- browser gate 页面能证明 PrimeVue adapter 已真实加载，而不是只显示静态 payload 文本。

### E2E-WIN-010：企业组件库路径选择成功，但真实企业安装未被验证

严重级别：P1
影响阶段：企业内置组件库场景

现象：solution confirmation 能正确选择 `enterprise-vue2 + enterprise-default`，但依赖安装在调用包管理器前即失败，未走到企业 registry 可达性或私有包凭证验证。

根因归纳：`WinError 2` 是更早层的命令执行问题，导致企业 registry URL、包名、认证、网络代理、私有包可用性全部未被测试。

解决方案：
- 先修 Windows 包管理器解析。
- 企业路径安装错误必须分类为命令不可执行、registry 不可达、代理问题、认证失败、包不存在、版本冲突。
- 企业 registry 前置条件应在 solution confirmation 或 apply preflight 中提前展示。

验收标准：
- 企业路径能至少执行到 npm registry 请求层。
- 私有包不可访问时返回明确认证/权限错误，不再返回 `[WinError 2]`。
- 成功路径记录企业包 manifest resolution 和 lockfile 证据。

### E2E-WIN-011：browser gate 与 baseline 能 fail-closed，但无法执行真实视觉验收

严重级别：P1
影响阶段：视觉测试验收

现象：`browser-gate-probe --execute` 返回 `apply_result_not_browser_gate_eligible`；`browser-gate-baseline` 返回 `browser_gate_artifact_missing`。这是结构化失败，但没有真实截图、visual regression bootstrap 或 baseline。

根因归纳：browser gate 依赖 managed delivery 成功和入口文件存在。当前 apply 失败后没有前端入口，也没有 Playwright runtime 证据，browser gate 只能 fail-closed。

解决方案：
- 修复依赖安装与代码生成后补跑 browser gate。
- browser gate 前置检查应列出缺失项：apply result、入口文件、runtime、baseline。
- baseline promotion 不应在缺少 bootstrap capture 时给出类似可继续的语义。

验收标准：
- 成功路径生成截图、bootstrap artifact、baseline png/yaml。
- baseline promotion 后二次 `browser-gate-probe --execute` 返回 pass 或明确 visual diff blocker。
- 失败路径继续保持 fail-closed，但 plain language blocker 可被普通用户理解。

### E2E-WIN-012：verify constraints 在前端交付失败后仍显示 no BLOCKERs

严重级别：P1
影响阶段：全局验证 / 收口信号

现象：在 managed delivery apply 失败、browser gate 缺失的状态下，`ai-sdlc verify constraints` 仍输出 `verify constraints: no BLOCKERs.`。

根因归纳：`verify constraints` 的职责边界偏仓库级规则/治理一致性，不等价于前端托管交付闭环验证。但当前绿色输出没有提醒用户“它不覆盖 managed frontend closure”。

解决方案：
- 在无 blocker 但存在 managed delivery failure artifact 时输出 advisory：不代表前端交付闭环通过。
- 或将关键 frontend evidence class 纳入 constraints 的可选 profile。

验收标准：
- 当前端 apply/browser gate/final proof 缺失时，`verify constraints` 不得单独制造“全项目已通过”的误解。
- 输出明确建议下一步运行 `program managed-delivery-apply` 或 `browser-gate-probe`。

### E2E-WIN-013：空目录最终收口对 Git 前置条件暴露过晚

严重级别：P2
影响阶段：最终收口

现象：按 README 前端闭环从空目录走到 `workitem close-check` 才提示 `git worktree list --porcelain failed ... not a git repository`，next action 为初始化 Git 并创建初始提交。

根因归纳：close-check 需要 Git inventory 和 git closure，但 README 最小前端闭环没有在早期声明 Git 是收口前置条件。

解决方案：
- 在 `init`、README 前端闭环、close-check preflight 中提前提示 Git 要求。
- 对无 Git 仓库场景提供降级说明：当前可继续做哪些只读/生成步骤，哪些 close 检查不可用。

验收标准：
- 空目录执行 `init` 或进入 frontend loop 时能看到 Git preflight warning。
- close-check 对无 Git 仓库返回清晰 blocker 和分步补救命令。

### E2E-WIN-014：Windows Git 默认分支 master 与 close-check 的 main 假设冲突

严重级别：P2
影响阶段：最终收口 / branch lifecycle

现象：`git init` 后默认分支为 `master`，提交成功后 close-check 执行 `git rev-list --left-right --count main...master`，因 `main` 不存在而 blocker。改为 `git branch -m main` 后该 blocker 消失。

根因归纳：`branch_inventory` 和 `git_client` 默认 base 为 `main`，没有从当前仓库自动识别默认分支，也没有在缺少 `main` 时给出针对性提示。

解决方案：
- 自动识别 default branch：优先 upstream HEAD，再看 `init.defaultBranch`，再看当前唯一初始分支。
- 或在项目 init 时明确创建/要求 `main`，并输出命令。

验收标准：
- 无 Git、`master`、`main`、自定义默认分支四种空项目状态都有测试覆盖。
- `master` 默认分支下不应误报为工作项 branch lifecycle drift；若强制 main，应给出 `git branch -m main`。

### E2E-WIN-015：Git 提交身份缺失时，PowerShell 串联命令仍继续执行后续 close-check

严重级别：P3
影响阶段：最终收口 / 用户操作指导

现象：`git commit` 因未配置 user.name/user.email 失败，但同一行后续 `ai-sdlc workitem close-check` 仍执行。

根因归纳：文档或操作示例使用 PowerShell 分号串联时，前一命令失败不会自动阻止后一命令执行。普通用户可能忽略 commit 失败而继续解读 close-check 结果。

解决方案：
- Windows 示例避免把 `git commit` 与 close-check 放在同一行。
- 如必须串联，使用显式 `$LASTEXITCODE` 检查或 PowerShell 7 的 `&&`。

验收标准：
- 文档中的 Windows close 示例按步骤执行。
- Git 身份缺失时，用户先看到身份配置修复命令，不会继续执行 close-check。

### E2E-WIN-016：workitem init 生成的 docs 在 Windows PowerShell 默认 Get-Content 下显示乱码

严重级别：P3
影响阶段：文档查看

现象：`Get-Content specs\001-order-ops-console\tasks.md` 显示乱码；`Get-Content -Encoding UTF8 ...` 显示正常。

根因归纳：PowerShell 5 默认编码行为与 UTF-8 文档不一致。文件内容本身可用，但普通 Windows 用户直接查看体验差。

解决方案：
- 文档提示使用 PowerShell 7 或 `Get-Content -Encoding UTF8`。
- 评估是否对 Windows 生成文档使用 UTF-8 BOM；需兼顾 macOS/Linux 和 Git diff。

验收标准：
- Windows 用户指南包含查看 UTF-8 文档的命令。
- 若采用 BOM 策略，测试覆盖 PowerShell 5/7、Python 读取、Git diff。

### E2E-WIN-017：final-proof-publication 在前端闭环失败时只给 warning，未形成收口 blocker

严重级别：P2
影响阶段：最终收口

现象：`program final-proof-publication --dry-run` 输出 `No frontend final proof publication steps required`，同时 warning `missing persisted write proof artifact`。

根因归纳：“无需发布”和“缺少前置 proof，无法发布”被同一个绿色语义覆盖。当前前端代码未生成、视觉验收未完成时，这种输出容易造成误解。

解决方案：
- final proof publication 区分 `not_applicable`、`blocked_missing_proof`、`ready_to_publish`。
- 缺少 persisted write proof 时，headline 不应是 `No steps required`。

验收标准：
- 当前端闭环失败时 final proof dry-run 返回明确 blocker 或至少非绿色 headline。
- 输出包含缺失 artifact 路径和恢复顺序：apply -> browser gate -> persisted write proof -> final proof。

### E2E-WIN-018：发布说明和用户文档对 V0.7.0 Windows 前端闭环表述过强

严重级别：P1
影响阶段：发布口径 / 用户预期

现象：`docs\releases\v0.7.0.md` 表述 `0.7.0` 聚焦从空项目到依赖安装、browser gate、baseline、close-stage readiness 的端到端闭环；README/USER_GUIDE 也写明 managed delivery 会写出最小受控前端产物并做真实依赖安装验证。但本轮 Windows E2E 只证明安装、初始化、solution confirmation 和 fail-closed，未证明真实依赖安装、前端代码落盘、视觉验收和最终 proof。

根因归纳：发布文档描述的是目标闭环或主干能力，但 release asset 的 Windows 普通路径没有完成相同证据级别的验证。

解决方案：
- 修复实现后补跑 Windows release asset E2E，再保留 release-grade 表述。
- 在修复前，发布说明应追加已知问题或撤回过强闭环承诺。
- 发布 checklist 增加“release asset 级 E2E 证据”，不能只引用源码仓库测试。

验收标准：
- release docs 中每个“已完成/已验证”声明都有对应 artifact、日志或测试命令证据。
- Windows offline zip、macOS/Linux 包分别有 release smoke 记录。
- 文档明确 `run --dry-run`、`verify constraints`、`solution confirmation` 各自不能替代完整前端闭环。

### E2E-WIN-019：企业场景未执行生成 payload 中混入公共 PrimeVue 命名

严重级别：P1
影响阶段：企业组件库代码生成 / provider 隔离

现象：企业场景 apply artifact 的 `artifact-generate` payload 中，provider、package、delivery entry 均为 `enterprise-vue2` / `@sxf/*`，但生成的 `src/generated/provider-adapter.ts` 和 `src/App.vue` 仍出现 `publicPrimeVueProviderComponents`、`publicPrimeVueProviderHelpers` 等公共 PrimeVue 命名。

根因归纳：企业 fallback scaffold 复用了公共 PrimeVue adapter 的符号命名。该 payload 因 dependency-install 失败未落盘，但在修复 P0 后会成为下一层 provider 隔离和代码可读性风险。

解决方案：
- provider adapter 模板按 effective provider 生成中性或企业命名，例如 `providerComponents`、`enterpriseProviderComponents`。
- 增加跨 provider payload snapshot 测试，禁止企业路径出现 `publicPrimeVue`、`PrimeVue` 等不相关符号。

验收标准：
- 企业路径生成 payload 中不出现公共 PrimeVue 专有符号。
- 公共路径仍保留 PrimeVue 真实 import 和命名。
- 成功落盘后 TypeScript/Vue 构建或静态检查通过。

### E2E-WIN-020：离线包不覆盖前端依赖闭环，离线/在线/企业 registry 模式未定义

严重级别：P0
影响阶段：离线交付 / dependency-install / 发布声明

现象：Windows 离线包能离线安装 Python/CLI，但 managed delivery 的公共路径仍需要公共 npm registry，企业路径仍需要 `http://npm.uedc.sangfor.com.cn/`，Playwright runtime 也需要额外下载或缓存。公共场景缺失 `pnpm` 后 fallback 到 `npm`，但 fallback 的风险、lockfile 口径和证据要求没有清晰说明。

根因归纳：“离线包”当前只覆盖 ai-sdlc CLI 安装，不等于前端依赖离线交付。前端依赖策略没有区分 `offline_strict`、`enterprise_registry`、`public_registry`，也没有在 mutate 前做模式级 preflight。

解决方案：
- 定义三种模式：`offline_strict`、`enterprise_registry`、`public_registry`。
- `offline_strict` 下如果缺少 npm/pnpm 缓存、包 tarball、Playwright browser runtime，应在 mutate 前 fail-fast。
- `enterprise_registry` 输出 registry URL、认证/代理前置、包存在性检查。
- `public_registry` 输出公共网络依赖、代理要求、lockfile 与 runtime 下载证据。

验收标准：
- 用户能从 CLI 输出和 artifact 判断当前前端依赖是否真正离线。
- strict 离线模式不访问公网或企业 registry；缺缓存时不进入部分写入。
- 在线/企业模式记录 registry 可达性、包存在性、lockfile、node_modules resolution、Playwright runtime 证据。

### E2E-WIN-021：managed delivery 失败 artifact 缺少 agent 可执行的恢复信息

严重级别：P1
影响阶段：错误诊断 / AI agent 自动恢复 / 用户手动恢复

现象：两个 apply artifact 在 P0 失败时均有 `remaining_blockers: ['[WinError 2] ...']`，但 `blocked_action_ids: []`、`plain_language_blockers: []`、`recommended_next_steps: []`。`visual-regression-runtime-install`、`artifact-generate`、公共场景 `workspace-integration` 因依赖失败未执行，却没有被标记为 dependency-blocked 或 skipped-with-reason。

根因归纳：ledger 只记录了直接失败 action，未把下游 action 的阻断关系、恢复入口和环境诊断写成结构化 contract。对 AI-Coding agent 来说，artifact 不足以自动判断下一步应检查 `npm.cmd`、PATH/PATHEXT、cwd、registry、fallback 或重试命令。

解决方案：
- dependency failure artifact 记录 attempted command、resolved executable、cwd、PATH/PATHEXT 摘要、package manager、registry、stdout/stderr、exception type、retry count。
- 下游 action 标记为 `dependency_blocked`，并记录 blocked-by action id。
- `plain_language_blockers` 和 `recommended_next_steps` 必须在 P0/P1 失败时非空。

验收标准：
- Windows `WinError 2` 时 artifact 能明确建议解析 `.cmd` 或安装/修复 package manager。
- agent 可根据 artifact 自动生成恢复命令并定位 working directory。
- browser gate/final proof 不再需要反向推断 apply 失败原因。

### E2E-WIN-022：遥测索引未反映 managed delivery/browser gate/final proof 未闭环

严重级别：P1
影响阶段：治理证据 / open violation / release 证据链

现象：E2E 项目 `.ai-sdlc\local\telemetry\indexes\latest-artifacts.json` 为 `artifact_ids: []`，`open-violations.json` 为 `violation_ids: []`，而实际 managed delivery、browser gate、final proof 都未闭环。现有 evidence 主要记录 `verify-constraints` passed。

根因归纳：前端托管交付失败没有进入 telemetry artifact/violation 索引，导致治理面看到的是“无开放违规”和“约束通过”，与用户路径失败相矛盾。

解决方案：
- managed delivery、browser gate、baseline、persisted write proof、final proof 的 latest artifact 应注册到 telemetry index。
- P0/P1 blocker 应形成 open violation 或 equivalent unresolved evidence item。
- `verify constraints` passed 不得覆盖其他 program surface 的未闭环状态。

验收标准：
- dependency-install 失败后 telemetry 能查询到对应 artifact id 和 open violation。
- 修复并重跑成功后 violation 被关闭或状态转为 resolved。
- release E2E 报告可从 telemetry 索引追溯每个关键 gate 的最终状态。

## 修复优先级建议

1. P0：修复 Windows `.cmd` 包管理器解析、前端依赖离线/在线/企业 registry 模式、依赖安装错误分类、代码生成执行顺序/摘要语义、业务 work item 模板隔离。
2. P1：补强 adapter 状态机与展示一致性、公共/企业组件库真实安装证据、失败 artifact agent 可执行性、遥测证据索引、browser gate 真实视觉验收、发布口径修正、企业 payload provider 隔离。
3. P2：补齐终端代理诊断、PRD 绑定说明、Git preflight、默认分支兼容、final proof 语义。
4. P3：修正文档查看编码提示和 PowerShell 串联命令示例。

## 回归测试建议

- Windows release asset 级 E2E：从 zip 解压开始，不使用源码仓库环境。
- Windows `subprocess` 包管理器单元测试：`npm.cmd`、`npx.cmd`、`pnpm.cmd`、缺失 pnpm fallback。
- 前端依赖模式测试：`offline_strict` 缺缓存 fail-fast、`enterprise_registry` 认证/包名分类、`public_registry` lockfile/runtime 证据。
- 两条 managed delivery 成功路径：企业 `enterprise-vue2`、公共 `public-primevue`。
- 失败分类路径：命令缺失、registry 不可达、认证失败、包不存在。
- 失败 artifact 合同：blocked downstream action、plain language blocker、recommended next steps、resolved command、cwd、stdout/stderr。
- telemetry 闭环：latest artifacts、open violations、resolved violations 与 program gate 状态一致。
- 业务 work item 生成：中文标题、中文输入、显式/隐式 `--wi-id`。
- adapter verified path：普通 PowerShell、Codex 环境变量、Cursor/IDE 内置终端、dry-run 不应意外物化或必须显式声明写入。
- close/final proof：无 Git、`master`、`main`、自定义默认分支、缺失 persisted write proof。

## 当前结论

本轮 V0.7.0 Windows 离线包可以完成安装和基础初始化；企业和公共 provider/style 的选择面可以落盘。但 Windows 普通用户路径无法完成真实组件库安装，导致前端代码生成、视觉测试验收和最终收口都不能闭环。最关键的发布级阻断包括：Windows 下包管理器执行失败 `[WinError 2]`、前端依赖离线/在线模式未定义、业务需求被错误分解为 framework direct-formal 模板、adapter 状态机与展示口径不一致、失败 artifact/telemetry 对 agent 不可执行，以及 release docs 对 Windows 前端闭环的表述强于 E2E 证据。

后续修复应优先保证 Windows 与 macOS/Linux 兼容，不应把命令简单改成只适配 PowerShell；所有发布声明都应回到 release asset 级 E2E 证据闭环。
