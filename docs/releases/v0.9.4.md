# AI-SDLC v0.9.4 Release Notes

## Positioning

`0.9.4` is a Vue3 frontend governance refresh release. It upgrades the
`public-primevue` default provider line from the previous Vue3 enterprise
standard to the v1.6.1 AI-executable contract.

This is not a provider switch. The ordinary default remains
`frontend_stack=vue3`, `provider_id=public-primevue`, and
`style_pack_id=modern-saas`. The release tightens how AI agents generate Vue3
enterprise UI after the user has confirmed the frontend solution.

## What Changed

### 1. Vue3 enterprise standard v1.6.1 is now the canonical frontend spec

- `docs/Vue3企业级前端开发规范方案.md` is now the v1.6.1 AI-executable standard.
- The spec keeps the default stack at Vue3, TypeScript, Vite, PrimeVue,
  `@primeuix/themes`, `primeicons`, UnoCSS, CSS Variables, Pinia, Vue Router,
  Axios, `vee-validate`, `zod`, and `vue-i18n`.
- It adds clearer project-boundary language so sample paths are not treated as
  mandatory global paths.
- It separates managed scaffold unit-test support from the ordinary default
  recommendation shown to users.

### 2. Frontend generation governance now matches v1.6.1

- Generation hard rules now include the v1.6.1 token floor, route meta contract,
  `ApiResponse<T>` contract, TypeScript unknown-first rule, theme token
  semantics, and dark-block boundary.
- Warning-only evidence such as missing `surface` / `highlight`, router meta,
  API response generics, and mixed formatting/business changes is reported as
  advisory warning evidence instead of blocking ordinary generation.
- `business-type-any` remains a blocker because it violates the TypeScript
  unknown-first boundary.
- PrimeVue `<InputText ...>` snippets are no longer misclassified as native
  `<input>` tags.

### 3. CLI and adapter guidance are aligned with the refreshed default

- `program solution-confirm` now presents the ordinary Vue3 public-primevue
  tooling without requiring Vitest in the user-facing default:
  `Playwright + ESLint + Prettier + husky + lint-staged + commitlint`.
- Managed scaffold and existing unit-test project paths can still use Vitest
  where that is the right project contract.
- AGENTS / Codex / Claude Code / VS Code / Cursor adapter templates use the
  same frontend confirmation wording.
- `verify constraints` now rejects stale default wording such as
  `Vitest + Playwright + ESLint + Prettier + husky + lint-staged + commitlint`
  when it appears in the ordinary default recommendation surface.

## Compatibility Notes

- This release does not remove the Vue2 compatibility provider or the advanced
  frontend solution-selection path.
- Vue3 `public-primevue` with `style_pack_id=modern-saas` remains the ordinary
  first recommendation for new frontend work.
- Explicit `vue2` / `enterprise-vue2` requests must still be honored.
- `ai-sdlc init .` remains the beginner setup entry. `adapter status` and
  `run --dry-run` remain troubleshooting commands, not required beginner-path
  setup steps.
- `python -m ai_sdlc ...` remains available as the module-entry fallback when a
  shell has not refreshed PATH yet.

## Install And Upgrade

Use the platform release asset from
`https://github.com/sinclairpan-git/Ai_AutoSDLC/releases/download/v0.9.4`.

- Windows: download `ai-sdlc-offline-0.9.4-windows-amd64.zip`, then run
  `install_offline.ps1 -AddToPath` for new installs or
  `install_offline.ps1 -UpgradeExisting` when replacing an older CLI.
- macOS / Linux: download the matching `.tar.gz` asset, unpack it, then run
  `install_offline.sh --add-to-path` for new installs or
  `--upgrade-existing` when replacing an older CLI.
- Existing users can try `ai-sdlc self-update check` first. If the output does
  not finish with version `0.9.4`, use the matching platform package and
  `--upgrade-existing`.
- If an older CLI reports `No such command 'install'`, skip diagnosis and use
  the matching platform package with `--upgrade-existing`.
