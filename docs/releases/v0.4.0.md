# AI-SDLC v0.4.0 Release Notes

## 版本定位

`0.4.0` 是一个能力面明显扩张的版本，核心变化不是小修小补，而是把前端治理主线从冻结设计正式推进到主线可用状态，并补齐一组会影响日常 CLI 使用稳定性的框架缺陷修复。

这次发布的重点是：

- 前端治理 MVP 主线正式合入主线
- Adapter 激活合同与持久化稳定性修复
- 文档安装/升级口径统一到 `v0.4.0`

## 本版本新增了什么

### 1. 前端治理 MVP 基线正式落地

主线已经正式合入 `009` 到 `018` 的前端治理 work item，形成了完整的 MVP 控制面：

- Frontend Contract authoring baseline
- Frontend Contract verify integration
- Frontend observation provider / scanner / export baseline
- Frontend runtime attachment baseline
- Frontend UI Kernel baseline
- `enterprise-vue2` Provider profile baseline
- Frontend generation governance baseline
- Frontend gate / compatibility baseline

对应能力已经进入主线代码，包括：

- `contracts/frontend/**` contract artifact 实例化
- `kernel/frontend/**` UI Kernel artifact 实例化
- `providers/frontend/enterprise-vue2/**` provider profile artifact 实例化
- `governance/frontend/generation/**` generation governance artifact 实例化
- frontend contract drift / observation / verification / gate helpers
- `verify constraints`、gate 聚合与 CLI summary 的前端接线
- frontend contract `scan --frontend-contract-spec-dir ...` export surface

这意味着框架已经具备前端 `Contract -> Kernel -> Provider -> Generation -> Gate` 的治理骨架。

### 2. `enterprise-vue2` Provider profile 进入主线真值

`enterprise-vue2` provider 现在已经有结构化 baseline，而不再只是设计稿描述：

- `Ui* -> 企业实现` 映射
- whitelist packaging 约束
- dangerous capability isolation
- legacy adapter boundary

当前版本里，这套 provider profile 的价值主要体现在“前端实现该如何受控地使用企业组件能力”，而不是直接提供完整业务页面 runtime。

### 3. 前端验证与运行时接线更完整

本版本补齐了从 observation export 到 verify/gate surface 的关键接线：

- canonical frontend observation loader
- runtime attachment helper
- `verify constraints` 的 frontend contract / frontend gate payload
- `VerificationGate / VerifyGate` 的前端摘要聚合
- CLI `verify constraints` 的前端摘要输出

这让前端实现不再只是“生成后人工看一遍”，而是可以进入结构化 verify / gate 路径。

## 本版本修复了什么问题

### 1. 修复 direct-formal workitem 入口在 bootstrap 缺口场景下的错误引导

修复前：

- 仓库里即使已有 `.ai-sdlc` 历史痕迹，只要缺 `project-state.yaml`
- `ai-sdlc workitem init` 就会先撞底层前置错误
- CLI 没有明确告诉用户应先执行 `ai-sdlc init .`

修复后：

- direct-formal 入口会明确提示 formal bootstrap 缺失
- 用户会被引导先执行 `ai-sdlc init .`

### 2. 修复 VS Code + Codex 场景下 adapter 激活识别偏差

修复前：

- 混合宿主场景可能把真正的 Codex 对话入口误识别成 VS Code 适配
- adapter 约束未必真正注入到实际聊天入口

修复后：

- agent target 识别与优先级口径已收紧
- adapter 激活合同与持久化更稳定

### 3. 修复 adapter activation UX 偏移

修复前：

- `init` 和 `adapter select` 过度依赖 `--agent-target`
- 交互式 selector 没有真正成为主路径

修复后：

- `init` 与 `adapter select` 共享交互式五项 selector
- TTY 场景下交互选择重新成为主 UX
- `--agent-target` 退回 non-interactive fallback / override

### 4. 修复 Windows 下 `project-config.yaml` 容易因句柄占用失败的问题

修复前：

- `YamlStore.save()` 采用临时文件 + `replace()`
- Windows 上目标文件短暂占锁时容易报 `PermissionError`
- adapter 元数据还会在“状态没变”的情况下重复改写 config

修复后：

- save 路径增加了 `serialize -> compare -> conditional write`
- 对 Windows `PermissionError` 增加 bounded retry / backoff
- adapter 只会在配置字段真实变化时刷新 `adapter_applied_at`

这显著降低了 Windows 上 `project-config.yaml` 的无意义重写和原子替换失败概率。

## 需要明确的边界

`0.4.0` 已经明显增强了前端治理能力，但它还不是“安装后直接生成成熟前端业务代码”的终局版本。

当前版本已经具备：

- 前端结构化 contract truth
- UI Kernel baseline
- `enterprise-vue2` provider profile baseline
- generation constraints
- gate / verify integration

当前版本仍未内置：

- 官方通用前端 runtime/provider pack
- 多 theme pack / 多风格开箱运行时
- 直接产出成熟业务页面源码的完整 codegen runtime

换句话说，`0.4.0` 交付的是“前端治理与验证底座”，不是“完整前端运行时产品”。
