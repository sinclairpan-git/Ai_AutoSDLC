# AI-SDLC v0.9.2 Release Notes

## Positioning

`0.9.2` is the Loop Engineering release. It moves AI-SDLC from a mostly linear
handoff process into five local, evidence-backed loops:

- Requirement loop
- Design-contract loop
- Implementation loop
- Frontend evidence loop
- Local adversarial PR review loop

The release is designed for mixed AI-SDLC users: beginners who need readable
commands and clear next actions, and professional developers or test engineers
who need deterministic artifacts, adaptable diff sources, and local model
control.

## What Changed

### 1. Five loop runtimes now have local artifacts and next actions

- `ai-sdlc loop requirement start/freeze` captures the original idea,
  acceptance criteria, clarification questions, and explicit freeze evidence
  before design work starts.
- `ai-sdlc loop design-contract check/close` validates that the frozen
  requirement is covered by `spec.md`, `plan.md`, and `tasks.md` before
  implementation starts.
- `ai-sdlc loop implementation start/record/close` tracks P0/P1 task progress,
  required evidence, and verification commands before downstream loops.
- `ai-sdlc loop frontend-evidence doctor/start/close/skip` consumes local
  browser-gate artifacts when available, reports missing browser capability in
  plain language, and allows an explicit audited skip when browser control
  cannot be installed or used.
- `ai-sdlc pr-review doctor/start/close/attest` supports local adversarial
  review through an independent provider command and writes review artifacts
  that can be read locally or by CI without asking CI to call a model service.

### 2. Review input and model selection are no longer hard-coded to GitHub

- PR review input is represented as a `DiffSource`, with support for local git
  ranges, local patches, staged changes, and unstaged changes.
- Future GitHub, GitLab, Gitee, self-hosted, or company-intranet SCM adapters
  can enter through source-provider adapters without changing the local review
  artifact model.
- The default review model selector is `current`, meaning the model already
  configured in the user's local CLI or agent environment.
- Advanced users can explicitly choose another connected model. If that model
  is unavailable, AI-SDLC reports the selected model as unavailable instead of
  silently falling back.

### 3. Frontend evidence is provider-first and cannot hard-block unsupported hosts

- `frontend-evidence doctor --provider auto` prefers existing evidence,
  Codex/browser-control capability, browser MCP/plugin control, or imported
  enterprise browser artifacts before suggesting Playwright.
- When Playwright is the chosen path, the doctor prints concrete package-manager
  install commands so users are not left to search for setup steps.
- If no browser provider can be used or installed, `frontend-evidence skip`
  records a risk-bearing skip artifact and moves the workflow to local PR
  review. A skip is not counted as browser-quality pass evidence.
- If browser evidence exists and reports blockers, `close --yes` still
  fail-closes until the blocker is fixed.

### 4. Loop status surfaces are safe to run anywhere

- `ai-sdlc loop status` and `ai-sdlc loop list` read local artifacts only.
- They do not start providers, call GPT, Claude, DeepSeek, GLM, Codex, or any
  other model service, fetch remote PR diffs, generate review packs, or modify
  application code.
- The commands show current state, unresolved counts, artifact paths, and the
  recommended next action in a beginner-readable form.

### 5. Release E2E evidence now covers the loop story

- `scripts/loop_e2e_release_gate.py` creates clean target repositories and
  executes the user-facing loop commands as subprocesses.
- The script writes raw stdout/stderr, JSON summaries, and terminal SVG captures
  that can be attached to release evidence.
- The GitHub workflow `loop-e2e-release-gate.yml` runs the loop E2E gate and can
  include the Windows Playwright-provider frontend evidence path.

## Compatibility Notes

- This release does not remove the Vue2 compatibility provider or the advanced
  frontend solution-selection path.
- Vue3 `public-primevue` with `style_pack_id=modern-saas` remains the ordinary
  first recommendation for new frontend work.
- `ai-sdlc init .` remains the beginner setup entry. `adapter status` and
  `run --dry-run` remain troubleshooting commands, not required beginner-path
  setup steps.
- `python -m ai_sdlc ...` remains available as the module-entry fallback when a
  shell has not refreshed PATH yet.
- CI may consume local review attestations and deterministic counts. CI should
  not be required to call model services from a restricted network.

## Install And Upgrade

Use the platform release asset from
`https://github.com/sinclairpan-git/Ai_AutoSDLC/releases/download/v0.9.2`.

- Windows: download `ai-sdlc-offline-0.9.2-windows-amd64.zip`, then run
  `install_offline.ps1 -AddToPath` for new installs or
  `install_offline.ps1 -UpgradeExisting` when replacing an older CLI.
- macOS / Linux: download the matching `.tar.gz` asset, unpack it, then run
  `install_offline.sh --add-to-path` for new installs or
  `--upgrade-existing` when replacing an older CLI.
- Existing users can try `ai-sdlc self-update check` first. If the output does
  not finish with version `0.9.2`, use the matching platform package and
  `--upgrade-existing`.
- If an older CLI reports `No such command 'install'`, skip diagnosis and use
  the matching platform package with `--upgrade-existing`.
