# AI-SDLC v0.5.0 Release Notes

## 版本定位

`0.5.0` 是一次面向主线可用性的收敛发布，重点不是扩一条全新产品线，而是把最近两轮已经进入主线的治理合同修正、文案真值对齐和 gate 稳定性改进正式收束成一个可发布版本。

这次发布的重点是：

- adapter governance activation state 正式收敛为可解释的三层状态面
- RefineGate 对验收场景标题的识别从窄格式放宽到常见 Markdown 写法
- backlog / USER_GUIDE / CLI / adapter 模板的真值口径再次对齐，并统一升级到 `v0.5.0`

## 本版本新增或强化了什么

### 1. adapter activation / governance state 口径正式分层

CLI、adapter 模板和用户文档现在统一区分：

- operator acknowledgement
- raw activation state
- derived governance mode

`adapter activate` 只会把当前文件型 adapter 推进到 `acknowledged`；`run --dry-run` 只代表安全预演成功；二者都不再被表述为“治理已激活”。

### 2. `adapter status` / `init` next-step 更明确

`init` 的 next-step 和 `adapter` 帮助文案现在会显式提示：

- 先 `adapter activate`
- 再用 `adapter status` 查看当前 adapter / governance state
- 文件型 adapter 仍是 `soft_prompt_only`
- operator acknowledgement 与 dry-run 都不是 governance proof

这让用户不再需要从零散文案里推断“文件已存在”是否等于“治理已生效”。

### 3. RefineGate 不再把常见 Markdown 场景标题误判为缺失

RefineGate 现在支持把以下形式识别为合法验收场景标题：

- `场景 1:`
- `Scenario 1:`
- `**场景 1**`
- `#### 场景 1`
- `- 场景 1`

也就是说，“每个用户故事必须有验收场景”继续是硬合同，但 gate 不再把它退化成单一排版陷阱。

## 本版本修复了什么问题

### 1. 修复 adapter activation 真值台账与重复指引段落漂移

修复前：

- backlog 顶部摘要可能与具体 defect 条目状态分叉
- `USER_GUIDE` 重复的 `adapter activate` 教程段落会漏掉 canonical disclaimer

修复后：

- backlog 总表与条目状态重新对齐
- `USER_GUIDE`、CLI、adapter 模板统一明确 `acknowledged` / `soft_prompt_only` / `not governance proof`

### 2. 修复 RefineGate 仅接受窄格式场景标题的误判

修复前：

- `**场景 1**`、`#### 场景 1`、`- 场景 1` 等常见 Markdown 写法会被误判为缺失验收场景

修复后：

- gate 会先做轻量 Markdown 归一化，再判断是否为场景标题
- 模板、规则文档和测试矩阵都与新合同保持一致

## 需要明确的边界

`0.5.0` 仍然没有为 Codex / Cursor / Claude Code / VS Code 这类文件型 adapter 引入宿主可验证 handshake。

当前版本已经具备：

- `installed / acknowledged / governance mode` 的显式状态面
- 一致的 CLI / 文档 / 模板口径
- 更稳健的 RefineGate 场景识别

当前版本仍未提供：

- 宿主侧可验证 activation signal
- 可以证明 IDE/agent 已硬采纳治理约束的 runtime handshake

换句话说，`0.5.0` 交付的是“更诚实、更稳定的治理状态表达”和“更少误判的 refine gate”，不是“宿主已实现硬接管”。
