# AI-SDLC v0.7.8 Release Notes

## Positioning

`0.7.8` is a production safety patch on top of `0.7.7`. It closes two defects found in real project use:

- `ai-sdlc recover --reconcile` could advance a direct-formal work item to execute even though `plan.md` and `tasks.md` were explicit placeholders.
- adapter target detection could prefer stale repo-local IDE marker directories over live AI host evidence, leaving projects stuck at the wrong adapter target or `materialized_unverified`.

The intended recovery behavior is conservative: existing files are candidate evidence, not proof that design, decompose, verify, or execute has completed.

The intended adapter behavior is runtime-led: live Codex / Claude Code / Cursor / VS Code host signals are more authoritative than historical `.cursor`, `.vscode`, `.codex`, or `.claude` directories in the repository.

The installed-user update path remains:

```bash
ai-sdlc self-update check
```

Legacy installs that report `No such command 'install'` should download the matching package from `releases/download/v0.7.8` and run `--upgrade-existing` once, then return to `ai-sdlc self-update check` for future updates.

## What Changed

### 1. Recover respects formal placeholder frontmatter

`recover --reconcile` now reads formal Markdown frontmatter before inferring completed stages:

- `stage: design-placeholder` stops reconciliation after `refine`; the next stage remains `design`
- `stage: decompose-placeholder` stops reconciliation after `design`; the next stage remains `decompose`
- execution-log files no longer pull placeholder direct-formal docs forward to `verify` or `execute`

### 2. Pipeline rules now state the recovery contract

The pipeline rule set now explicitly says file existence is not stage completion. Recovery and resume paths must respect frontmatter, gates, and verifiable evidence when rebuilding checkpoint state.

### 3. Adapter target detection follows the live AI host

Adapter detection now prefers live runtime evidence such as `OPENAI_CODEX` or `CLAUDE_CODE_ENTRYPOINT` before falling back to repo-local marker directories. This prevents a project that contains old `.cursor` or `.vscode` folders from selecting the wrong adapter when the current chat host is Codex or Claude Code.

`adapter status` also exposes target mismatches more clearly: when the detected live host differs from the configured `agent_target`, the beginner guidance points users to `ai-sdlc adapter select --agent-target <detected-host>`.

The status surface remains tolerant of legacy or custom persisted `detected_ide` strings. Unknown persisted values now fall back to live detection instead of crashing `adapter status`, `run`, or other callers of the adapter governance surface.

### 4. Regression coverage locks the defects

The regression suite now covers the recover production-shaped scenario:

- `spec.md` exists
- `plan.md` is marked `stage: design-placeholder`
- `tasks.md` is marked `stage: decompose-placeholder`
- `task-execution-log.md` exists
- after `recover --reconcile`, checkpoint stays at `design` with only `init` and `refine` recorded as complete

An additional unit case covers a real plan with a decompose placeholder and keeps the next stage at `decompose`.

The adapter regression suite now covers:

- live Codex and Claude Code runtime evidence winning over stale repo IDE markers
- `init` selecting Codex when `OPENAI_CODEX` is present even if `.cursor/` exists
- `adapter status` surfacing target mismatch guidance when a project initialized for Cursor is later run from Codex
- unknown persisted `detected_ide` values falling back to live detection instead of raising `ValueError`

## Compatibility Notes

- macOS / Linux use `.tar.gz` platform packages:
  - macOS Apple Silicon uses `ai-sdlc-offline-0.7.8-macos-arm64.tar.gz`
  - Linux x64 uses `ai-sdlc-offline-0.7.8-linux-amd64.tar.gz`
- Windows x64 uses `ai-sdlc-offline-0.7.8-windows-amd64.zip`
- The update path from installed releases remains `ai-sdlc self-update check`
- Source checkouts and editable installs remain development environments; release installers should not overwrite them silently
