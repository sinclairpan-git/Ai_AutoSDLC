# AI-SDLC v0.7.1 Release Notes

## Legacy Update Notice

If you are reading this page while trying to upgrade an older installed CLI, do
not follow this historical release's asset names or update behavior. Go to the
latest release and use the platform install package's upgrade mode:

- Latest release: `https://github.com/sinclairpan-git/Ai_AutoSDLC/releases/tag/v0.7.10`
- Download and extract the matching package first:
  - macOS Apple Silicon: `ai-sdlc-offline-0.7.10-macos-arm64.tar.gz`
  - Linux x64: `ai-sdlc-offline-0.7.10-linux-amd64.tar.gz`
  - Windows x64: `ai-sdlc-offline-0.7.10-windows-amd64.zip`
- Then enter the extracted folder and run:
  - macOS / Linux: `./install_offline.sh --upgrade-existing`
  - Windows: `powershell -ExecutionPolicy Bypass -File .\install_offline.ps1 -UpgradeExisting`

After that one-time bridge, future updates should use `ai-sdlc self-update check`.

## Release Positioning

`0.7.1` is a release-path hardening patch on top of `0.7.0`. It keeps the frontend managed-delivery contract from `0.7.0`, and focuses on ensuring the offline smoke and release artifact verification story is current, repeatable, and auditable.

## What This Release Delivers

### 1. Windows offline smoke no longer uses the deprecated Node 20 action path

- `.github/workflows/windows-offline-smoke.yml` now uploads smoke evidence with `actions/upload-artifact@v7`
- This removes the `actions/upload-artifact@v4` Node 20 runtime warning from the offline smoke workflow while keeping the same evidence artifact contract

### 2. Published release assets have their own smoke workflow

- `.github/workflows/release-artifact-smoke.yml` can be run after publishing `v0.7.1`
- The workflow downloads `ai-sdlc-offline-0.7.1.zip` and `ai-sdlc-offline-0.7.1.tar.gz` from GitHub Release assets
- It installs the downloaded bundles and runs `ai-sdlc --help`, `ai-sdlc adapter status`, and `ai-sdlc run --dry-run`

### 3. Cloud release build can produce platform-specific assets

- `.github/workflows/release-build.yml` builds on `windows-latest`, `macos-latest`, and `ubuntu-latest`
- Each matrix job builds the offline bundle on its own runner, runs the matching installer smoke, and only then uploads the smoke-passed asset to the GitHub Release
- Cloud-built assets use platform suffixes such as `ai-sdlc-offline-0.7.1-windows-amd64.zip`, `ai-sdlc-offline-0.7.1-macos-arm64.tar.gz`, or `ai-sdlc-offline-0.7.1-linux-amd64.tar.gz`

### 4. Release docs and version truth are aligned

- `README.md`, `USER_GUIDE.zh-CN.md`, `packaging/offline/README.md`, and `docs/pull-request-checklist.zh.md` now point at `v0.7.1`
- Release consistency checks now require `docs/releases/v0.7.1.md` and the `0.7.1` offline asset names

## Boundaries

- `release-artifact-smoke.yml` validates assets that already exist on the GitHub Release; it does not build or upload the official release assets
- Windows `.zip` and POSIX `.tar.gz` smoke evidence remains platform-scoped; a successful archive build is not the same as target-platform install evidence
- Enterprise private frontend registry verification remains outside this patch unless the required registry credentials are available to the workflow runner

## Release Verification

Required before marking `v0.7.1` complete:

- `uv run pytest tests/integration/test_github_workflows.py tests/integration/test_offline_bundle_scripts.py tests/unit/test_packaging_backend.py tests/unit/test_verify_constraints.py -q`
- `uv run ruff check src tests`
- `uv run ai-sdlc verify constraints`
- `python -m ai_sdlc program truth sync --dry-run`
- To build and upload release assets from cloud runners: `release-build.yml` for tag `v0.7.1`
- After publishing the GitHub Release assets: `release-artifact-smoke.yml` for tag `v0.7.1`
