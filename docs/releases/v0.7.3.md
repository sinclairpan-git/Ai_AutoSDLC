# AI-SDLC v0.7.3 Release Notes

## Legacy Update Notice

If you are reading this page while trying to upgrade an older installed CLI, do
not follow this historical release's asset names or update behavior. Go to the
latest release and use the platform install package's upgrade mode:

- Latest release: `https://github.com/sinclairpan-git/Ai_AutoSDLC/releases/tag/v0.7.9`
- Download and extract the matching package first:
  - macOS Apple Silicon: `ai-sdlc-offline-0.7.9-macos-arm64.tar.gz`
  - Linux x64: `ai-sdlc-offline-0.7.9-linux-amd64.tar.gz`
  - Windows x64: `ai-sdlc-offline-0.7.9-windows-amd64.zip`
- Then enter the extracted folder and run:
  - macOS / Linux: `./install_offline.sh --upgrade-existing`
  - Windows: `powershell -ExecutionPolicy Bypass -File .\install_offline.ps1 -UpgradeExisting`

After that one-time bridge, future updates should use `ai-sdlc self-update check`.

## Release Positioning

`0.7.3` is a beginner CLI onboarding patch on top of `0.7.2`. It keeps the installed-runtime update advisor and release asset workflow from `0.7.2`, then tightens the first-run experience so a non-technical user can finish initialization from the CLI output without translating internal framework state.

## What This Release Delivers

### 1. `init` now completes the safe first-run path

- `ai-sdlc init` still asks for necessary human choices such as AI agent target and project shell
- After those choices, `init` automatically performs the adapter/status and safe rehearsal path instead of asking the user to copy several follow-up commands
- Successful setup ends with a bilingual result telling the user to switch to Codex/AI chat and describe the requirement
- If the automatic rehearsal crashes, initialization fails loudly and gives a single recovery command instead of claiming setup is complete

### 2. Beginner-safe default CLI output

- Default `adapter status`, `host-runtime plan`, and `run --dry-run` output now focuses on: is this normal, what happened, and the single next command
- Internal governance fields, digests, and activation details remain available through JSON/detail-oriented paths instead of crowding the default user view
- Guidance is bilingual Chinese/English for the result and next action

### 3. Cleaner recovery guidance

- Supported agent targets no longer suggest generic or misleading environment-variable workarounds
- Generic shells guide users back to adapter selection instead of pretending generic ingress can be verified as a real AI host
- Mutating runs still stop when host ingress is unverified, but the recovery message is now action-oriented

### 4. Release docs and workflow defaults are aligned

- `README.md`, `USER_GUIDE.zh-CN.md`, `packaging/offline/README.md`, `docs/pull-request-checklist.zh.md`, and `docs/框架自迭代开发与发布约定.md` now point at `v0.7.3`
- `release-build.yml` and `release-artifact-smoke.yml` default to `v0.7.3`
- Release consistency checks now require `docs/releases/v0.7.3.md` and the `0.7.3` offline asset names

## Boundaries

- `run --dry-run` remains a safe rehearsal and does not by itself prove machine-verifiable host ingress
- Advanced adapter truth is still available for maintainers through `adapter status --json`
- Offline release assets remain platform-scoped: Windows `.zip`, macOS / Linux `.tar.gz`
