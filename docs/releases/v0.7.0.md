# AI-SDLC v0.7.0 Release Notes

## Legacy Update Notice

If you are reading this page while trying to upgrade an older installed CLI, do
not follow this historical release's asset names or update behavior. Go to the
latest release and use the platform install package's upgrade mode:

- Latest release: `https://github.com/sinclairpan-git/Ai_AutoSDLC/releases/tag/v0.7.14`
- Download and extract the matching package first:
  - macOS Apple Silicon: `ai-sdlc-offline-0.7.14-macos-arm64.tar.gz`
  - Linux x64: `ai-sdlc-offline-0.7.14-linux-amd64.tar.gz`
  - Windows x64: `ai-sdlc-offline-0.7.14-windows-amd64.zip`
- Then enter the extracted folder and run:
  - macOS / Linux: `./install_offline.sh --upgrade-existing`
  - Windows: `powershell -ExecutionPolicy Bypass -File .\install_offline.ps1 -UpgradeExisting`

After that one-time bridge, future updates should use `ai-sdlc self-update check`.

## Release Positioning

`0.7.0` is the release that turns the recent frontend managed-delivery closure work from "main branch capability" into a published framework contract. Compared with `0.6.0`, this version is focused on making the governed frontend path actually close end to end from an empty project through dependency installation, browser-gate execution, baseline promotion, and close-stage readiness.

## What This Release Delivers

### 1. Empty-project frontend bootstrap is now release-grade

- `workitem init` can bootstrap a minimal `program-manifest.yaml` when the target project has not established program truth yet
- The frontend solution confirmation and managed-delivery path can therefore start from a real cold-start project instead of assuming pre-existing manifest truth

### 2. Managed delivery now performs real dependency installation verification

- `program managed-delivery-apply --execute --yes` no longer stops at apply artifacts alone
- The command now verifies package installation through package manifest, lockfile, and dependency-resolution evidence
- Browser-gate runtime prerequisites such as Playwright browser runtime can be auto-installed as part of the managed frontend closure loop

### 3. Browser-gate baseline promotion is now a formal CLI surface

- `program browser-gate-baseline --dry-run`
- `program browser-gate-baseline --execute --yes`

These commands promote the latest `visual_regression_bootstrap` capture into a governed baseline under:

- `governance/frontend/quality-platform/evidence/visual-regression/baselines/<matrix-id>/baseline.png`
- `governance/frontend/quality-platform/evidence/visual-regression/baselines/<matrix-id>/baseline.yaml`

### 4. Browser-gate artifact handling is hardened to fail closed

This release closes several artifact-driven safety gaps:

- non-mapping browser-gate YAML roots no longer crash baseline preparation
- invalid bootstrap artifact refs no longer crash execution
- traversal-style `matrix_id` values can no longer escape the intended baselines directory
- malformed artifact-derived paths now return structured blockers instead of uncaught exceptions

### 5. Release-facing documentation now matches the shipped command surface

The published entry docs have been updated so users can follow the current release path consistently across:

- `README.md`
- `USER_GUIDE.zh-CN.md`
- `packaging/offline/README.md`
- `packaging/offline/RELEASE_CHECKLIST.md`

## Value Delivered by This Release

- A user can start from a colder project state without hand-creating `program-manifest.yaml`
- Component-library installation has machine-verifiable evidence instead of artifact-only claims
- Browser-gate closure has a formal baseline promotion step instead of an ad hoc internal workaround
- Artifact-driven frontend quality surfaces fail closed instead of crashing on malformed input
- Published docs now explain the real released command loop instead of describing these capabilities as unreleased-only behavior

## Boundaries

`0.7.0` strengthens the release path, but it does not claim that every environment is fully automatic.

Known boundaries that still depend on host conditions:

- real browser execution still depends on the target machine being able to launch the selected browser runtime
- offline bundle release claims still require per-platform smoke evidence
- enterprise private registries still require reachable registry endpoints and correct package names / credentials
