# AI-SDLC v0.5.1 Release Notes

## 版本定位

`0.5.1` 是一次生产故障修复型补丁发布，目标很单一：把已安装包场景下 `workitem init` 因模板文件缺失而失败的问题彻底修正，并把这条运行时路径纳入真实安装态回归。

这次发布的重点是：

- 修复 installed-wheel runtime 下 `workitem init` 找不到 `spec/plan/tasks/execution-log` 模板的问题
- 把 Markdown 模板正式打入 wheel，避免“源码态正常、安装态失效”
- 为真实 installed-wheel CLI 路径补上自动化 smoke，防止同类 defect 再次漏过发布

## 本版本修复了什么问题

### 1. 修复 `workitem init` 在已安装包场景下的模板查找失败

修复前：

- `WorkitemScaffolder` 主要依赖源码仓根目录 `templates/`
- 已安装 wheel 运行时缺少对应 Markdown 模板资源
- 用户在生产安装环境中执行 `ai-sdlc workitem init` 会直接失败

修复后：

- 模板查找顺序收敛为：显式 `template_dir` -> 仓库根 `templates/` -> 包内 `ai_sdlc/templates/`
- `spec-template.md`、`plan-template.md`、`tasks-template.md`、`execution-log-template.md` 均随 wheel 分发
- 安装态 CLI 可以直接生成 work item 基础文件，不再依赖源码仓布局

### 2. 修复发布验证只覆盖源码态、未覆盖真实安装态的问题

修复前：

- 测试主要覆盖 source-tree `CliRunner` 和局部 fallback
- wheel 内容检查不能替代真实安装后的 CLI 冒烟
- 生产安装路径仍可能在发布前漏检

修复后：

- 新增隔离 `python3.11` 环境下的 installed-wheel smoke
- 测试会先 `uv build`，再安装当前 wheel，并用安装后的 `ai-sdlc` 真实执行 `init` / `workitem init`
- 同时锁住 wheel 打包配置和模板 fallback 逻辑

## 需要明确的边界

`0.5.1` 是补丁版，不引入新的产品面，也不改变既有治理策略。它只修复一个已经在生产暴露的安装态运行时缺陷，并补强对应的发布防回归证据链。

换句话说，`0.5.1` 交付的是“安装态 workitem scaffolding 重新可用”和“发布前真实安装态 smoke 已补齐”，不是新的 CLI 功能扩展。
