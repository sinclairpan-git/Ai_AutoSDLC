# 代码审查协议

> 本规则在 EXECUTE 阶段的每个批次提交前激活。每次 git commit 之前，AI 必须对本批次的代码变更执行自审查。
> **跳过审查 = 跳过质量。没有例外。**

## 铁律

```
没有审查通过，就不能 git commit。
```

## 何时执行

**必须执行（强制）：**
- 每个批次 commit 前
- 主要功能实现完成后
- 复杂 bug 修复后

**建议执行：**
- 重构完成后
- 卡住时（换个视角审视）
- 批次间的集成点

## 审查维度

每次审查必须覆盖以下 6 个维度，按优先级排序：

### 维度 1：宪章对齐（Critical）

```
检查项：
  □ 本批次的实现是否偏离 constitution.md 中的核心原则？
  □ 是否违反了技术硬约束？
  □ 是否引入了宪章「非目标清单」中的功能？
  □ 改动范围是否超出了当前任务的声明范围？

判断标准：
  任何一项不通过 → 必须修复后才能 commit
```

### 维度 2：需求规格对齐（Critical）

```
检查项：
  □ 实现是否完整覆盖了 spec.md 中对应 US 的验收标准？
  □ 实现的行为是否与 spec.md 中的 Given-When-Then 场景一致？
  □ 是否引入了 spec 中未描述的额外行为？
  □ 术语使用是否与 spec 一致（无术语漂移）？

判断标准：
  覆盖缺口 → 标记为 GAP，记录到 execution-log
  术语不一致 → 统一后再 commit
  额外行为 → 评估是否需要（记录为 AI 决策）
```

### 维度 3：技术规范一致性（Important）

```
检查项：
  □ 代码是否符合 tech-stack.yml 中定义的编码规范？
  □ 命名是否遵循项目约定（驼峰/下划线/文件名规则）？
  □ 目录结构是否符合 plan.md 中的项目结构定义？
  □ API 实现是否与 contracts/ 中的契约一致？
  □ 数据库操作是否与 data-model.md 一致？

判断标准：
  不一致 → 修复后再 commit
```

### 维度 4：代码质量（Important）

```
检查项：
  □ 无重复代码（如发现 → 提取公共函数）
  □ 函数/方法长度合理（单一职责）
  □ 错误处理完整（不吞异常、不忽略错误返回值）
  □ 无硬编码的魔法数字/字符串
  □ 无安全漏洞（SQL 注入、XSS、未校验输入）
  □ 无性能隐患（N+1 查询、无限循环风险、内存泄漏）
  □ 日志有意义（不过多不过少）

判断标准：
  安全漏洞 → 必须修复（Critical）
  其他 → 视严重程度决定（记录到 execution-log）
```

### 维度 5：测试质量（Important）

```
检查项：
  □ 测试是否遵循 TDD 规则（见 rules/tdd.md）？
  □ 测试覆盖了 happy path + 边界 + 错误路径？
  □ 测试命名遵循 should_X_when_Y 规范？
  □ 没有测 mock 行为而忽略真实行为的测试？
  □ 测试之间无顺序依赖？

判断标准：
  缺失边界测试 → WARNING（记录但不阻断）
  测试反模式 → 修复后再 commit
```

### 维度 6：Spec 偏移检测（Important）

```
检查项：
  □ 实现过程中是否发现 spec 有歧义或不完整？
  □ 实际实现是否与原始设计产生了偏移？
  □ 偏移是否已记录为 AI 决策（带理由）？

判断标准：
  偏移未记录 → 补充 AI 决策记录后再 commit
  重大偏移 → 评估是否需要回溯更新 spec
```

## 审查输出格式

审查结果记录在 execution-log 的批次记录中：

```markdown
#### 代码审查 | Batch N

**审查范围**：[文件列表]

| 维度 | 结果 | 问题 |
|------|------|------|
| 宪章对齐 | ✅ PASS | — |
| 规格对齐 | ⚠️ GAP | FR-003 覆盖不完整 |
| 技术规范 | ✅ PASS | — |
| 代码质量 | ✅ PASS | — |
| 测试质量 | ⚠️ WARNING | 缺少超时边界测试 |
| Spec 偏移 | ✅ PASS | — |

**需修复**：
- [ ] 补充 FR-003 的邮箱格式校验逻辑

**已记录**：
- WARNING: 缺少超时边界测试（不阻断，记入 execution-log）
```

## 判定规则

```
所有维度 PASS 或仅 WARNING → 可以 commit
任何维度有需修复项 → 修复后重新审查（不需要重审所有维度，只审改动相关维度）
修复次数 > 2 → 评估是否任务拆分有问题
```

## 与批次协议的集成

在 `batch-protocol.md` 的 Step 3（验证）和 Step 5（提交）之间插入审查步骤：

```
Step 3：验证 → 全部通过
Step 3.5：代码审查（本规则）→ 全部通过或仅 WARNING
Step 4：归档
Step 5：提交
```

## 不是什么

本审查协议**不是**：
- 代码美化 — 不做纯风格偏好的修改
- 完美主义 — WARNING 记录但不阻断
- 形式主义 — 审查要快速（< 2 分钟思考），重点是安全、正确性、一致性

## 快速参考

| 严重级 | 处理方式 | 示例 |
|--------|---------|------|
| **Critical** | 必须修复再 commit | 安全漏洞、宪章违反、spec 覆盖缺口 |
| **Important** | 应修复，可酌情记录 | 命名不一致、缺少错误处理 |
| **WARNING** | 记录到 log，不阻断 | 缺少边界测试、可优化但不影响正确性 |
