# TDD 纪律

> 本规则在 EXECUTE 阶段激活，强制测试先行。
> **违反本规则的字面要求即违反其精神。**

## 铁律

```
没有失败的测试，就不能写生产代码。
```

先写了实现代码？删掉。从测试开始重写。

**没有例外：**
- 不保留作"参考"
- 不"改编"已有实现
- 不看它
- 删除就是删除

从测试出发，全新实现。句号。

## 何时执行

**必须执行：**
- 新功能
- Bug 修复
- 重构
- 行为变更

**可豁免（需用户确认）：**
- 一次性原型
- 纯配置文件
- 纯文档任务
- 纯前端样式（无业务逻辑）
- tasks.md 中该批次未标注测试任务

即使豁免 TDD，批次结束后仍必须运行全量回归。

想着"就这一次跳过 TDD"？停下来。那是自我合理化。

## Red-Green-Verify 循环

### RED — 写失败测试

写一个最小测试，描述预期行为。

**好的测试：**
```typescript
test('连续失败 3 次后触发重试上限异常', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('fail');
    return 'success';
  };

  const result = await retryOperation(operation);
  expect(result).toBe('success');
  expect(attempts).toBe(3);
});
```
名称清晰、测试真实行为、只测一件事。

**差的测试：**
```typescript
test('retry works', async () => {
  const mock = jest.fn()
    .mockRejectedValueOnce(new Error())
    .mockResolvedValueOnce('success');
  await retryOperation(mock);
  expect(mock).toHaveBeenCalledTimes(2);
});
```
名称含糊、测 mock 而非真实代码。

**测试命名规范：**

```
命名模板：should_[预期行为]_when_[条件]
示例：
  ✅ should_reject_with_validation_error_when_email_is_empty
  ✅ should_retry_3_times_when_operation_fails
  ❌ test1
  ❌ retry works
  ❌ email validation test
```

**测试要求：**
- 一个行为 — "and" 出现在名称中？拆分
- 清晰命名 — 描述行为，不是描述实现
- 真实代码 — 除非不可避免，不用 mock
- 名称中包含"and"？说明在测多件事，必须拆分

### 验证 RED — 看它失败

**必须执行。绝不跳过。**

```bash
[项目测试命令] path/to/test.test.ts
```

确认：
- 测试**失败**（不是报错/error）
- 失败信息符合预期
- 失败原因是"功能未实现"，不是拼写错误、导入错误

**测试通过了？** 你在测已有行为。修改测试。
**测试报错了？** 修复错误，重跑直到正确失败。

### GREEN — 最小实现

写通过测试的最简代码。

**好的实现：**
```typescript
async function retryOperation<T>(fn: () => Promise<T>): Promise<T> {
  for (let i = 0; i < 3; i++) {
    try { return await fn(); }
    catch (e) { if (i === 2) throw e; }
  }
  throw new Error('unreachable');
}
```
刚好通过测试。

**差的实现：**
```typescript
async function retryOperation<T>(
  fn: () => Promise<T>,
  options?: {
    maxRetries?: number;
    backoff?: 'linear' | 'exponential';
    onRetry?: (attempt: number) => void;
  }
): Promise<T> { /* YAGNI */ }
```
过度设计，测试没要求这些。

不要添加功能、重构其他代码、或做测试范围之外的"改进"。

### 验证 GREEN — 看它通过

**必须执行。**

```bash
[项目测试命令] path/to/test.test.ts
```

确认：
- 目标测试通过
- 其他测试仍然通过
- 输出干净（无 error、warning）

**测试失败？** 修实现代码，不是修测试。
**其他测试失败了？** 立即修复。

### REFACTOR — 清理

仅在 GREEN 之后：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试绿灯。不加新行为。

### 循环

下一个失败测试 → 下一个功能。

## 测试质量标准

| 维度 | 要求 | 反例 |
|------|------|------|
| **最小化** | 一件事。名称里有"and"？拆分 | `test('验证邮箱和域名和空格')` |
| **清晰** | 名称描述行为 | `test('test1')` |
| **意图** | 展示期望的 API 用法 | 测试内部实现细节 |
| **确定性** | 每次运行结果一致 | 依赖时间、网络、随机数 |
| **隔离** | 不依赖其他测试的执行顺序或状态 | 共享全局变量、依赖测试执行顺序 |

## 边界覆盖强制要求

每个用户故事的测试必须覆盖以下边界（如适用）：

```
必测边界：
  ├── 空值 / null / undefined / 空字符串
  ├── 边界值（0, -1, MAX_INT, 空数组, 单元素数组）
  ├── 错误路径（不能只测 happy path）
  ├── 并发场景（如功能涉及并发）
  ├── 超时 / 网络异常（如功能涉及外部调用）
  └── 权限边界（如功能涉及鉴权）

可选但建议：
  ├── 大数据量 / 性能边界
  └── 状态转换的非法路径（如订单从"已发货"直接到"未支付"）
```

缺少边界测试的批次，门禁检查时标记为 WARNING，不阻断但记录在 execution-log 中。

## 测试元检查

每个测试写完后，自问以下 3 个问题（不需要输出，但必须在思考中完成）：

```
Q1：如果我删掉实现中的关键一行代码，这个测试会失败吗？
    → 如果不会，测试没有测到关键行为

Q2：如果我写了一个错误的实现（返回硬编码值），这个测试能发现吗？
    → 如果不能，测试不够具体

Q3：这个测试 6 个月后还能读懂吗？
    → 如果读不懂，改善命名和结构
```

## 测试反模式防御

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 测 mock 行为而非真实行为 | mock 通过但真实代码崩溃 | mock 仅用于不可控外部依赖 |
| 给生产类添加 test-only 方法 | 污染生产代码 | 通过公开 API 测试 |
| 不理解依赖就 mock | 掩盖集成问题 | 先理解依赖行为再决定是否 mock |
| 测试依赖执行顺序 | 单独跑通整体跑挂 | 每个测试独立 setup/teardown |
| 断言只检查"不抛异常" | 没有验证正确性 | 断言具体的返回值和状态变更 |
| 快照测试替代行为测试 | 快照变了不知道是 bug 还是需求变更 | 快照作为补充，不替代行为断言 |

## 执行协议（与批次协议集成）

对每个用户故事批次：

```
步骤 1：执行该批次的所有测试任务（T0XX [P] [USN] 新增...测试）
步骤 2：运行这些测试 → 记录为 R1（Red 验证）
步骤 3：确认测试失败且失败原因是"功能未实现"（而非测试本身有 bug）
步骤 4：执行该批次的所有实现任务
步骤 5：运行同一组测试 → 记录为 V1（Green 验证）
步骤 6：如果 V1 有失败：
  - 分析失败原因（遵循「调试协议」rules/debugging.md）
  - 修复实现（不是修改测试来绕过）
  - 重新运行（最多 3 轮）
步骤 7：运行全量回归 → 记录为 V2
步骤 8：V2 通过 → 进入批次内自审查（见 rules/code-review.md）
```

## 记录格式

在 task-execution-log.md 的批次记录中，必须包含：

```markdown
- `R1`（红灯验证）
  - 命令：[实际执行的测试命令]
  - 结果：按 TDD 预期失败，[N] 个断言失败，原因：[功能未实现]
  - 边界覆盖：[列出本批次覆盖的边界类型]
- `V1`（绿灯验证）
  - 命令：[同一测试命令]
  - 结果：成功，Tests run: N, Failures: 0
- `V2`（全量回归）
  - 命令：[全量测试命令]
  - 结果：成功，Tests run: N, Failures: 0
```

## Bug 修复的 TDD

发现 bug？先写复现测试。遵循 TDD 循环。测试既证明修复有效，又防止回归。

**绝不在没有测试的情况下修 bug。**

## 为什么顺序重要

**"我先写代码再补测试验证"**

后补测试立即通过。立即通过什么都不证明：
- 可能测的是错误的东西
- 可能测的是实现细节而非行为
- 可能遗漏了你忘记的边界
- 你从没看到它捕获过 bug

先写测试逼你看到它失败，证明它确实在测东西。

**"删掉 X 小时的工作太浪费了"**

沉没成本谬误。时间已经过去了。你现在的选择：
- 删除并用 TDD 重写（X 小时，高信心）
- 保留并补测试（30 分钟，低信心，大概率有 bug）

"浪费"的是保留你无法信任的代码。

## 自我合理化防御表

| 借口 | 现实 |
|------|------|
| "太简单不用测" | 简单代码也会坏。测试 30 秒的事。 |
| "先写后补测试一样" | 后补 = "这做了什么？" 先写 = "这该做什么？" |
| "已经手动测过了" | 手动 ≠ 系统化。无记录、不可重复。 |
| "删掉 X 小时太浪费" | 沉没成本谬误。保留未验证代码才是技术债。 |
| "保留做参考，测试先写" | 你会"改编"它。那就是后补测试。删除就是删除。 |
| "需要先探索" | 可以。探索完扔掉，用 TDD 重写。 |
| "测试难写 = 设计有问题" | 听测试的。难测 = 难用。简化接口。 |
| "TDD 会拖慢速度" | TDD 比 debug 快。先测试 = 真正的务实。 |
| "这次情况不一样" | 不一样。每次都不一样。但 TDD 不变。 |

## 红旗 — 立即停下并重新开始

- 先写了代码再补测试
- 测试立即通过（没看到红灯）
- 无法解释测试为什么失败
- "之后再补测试"
- 在合理化"就这一次"
- "保留做参考"或"改编现有代码"
- "太浪费了不想删"
- "这次不同因为..."

**以上全部意味着：删除代码。用 TDD 重写。**

## 被卡住时

| 问题 | 解决方案 |
|------|---------|
| 不知道怎么测 | 先写你期望的 API 调用方式。先写断言。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 必须 mock 所有东西 | 耦合太紧。用依赖注入。 |
| 测试 setup 太大 | 提取 helper。仍然复杂？简化设计。 |

## 完成前检查清单

标记批次完成前，逐项确认：

- [ ] 每个新函数/方法都有测试
- [ ] 看到了每个测试失败（红灯验证）
- [ ] 每个测试失败的原因正确（功能未实现，不是拼写错误）
- [ ] 写了最小代码通过每个测试
- [ ] 所有测试通过
- [ ] 输出干净（无 error、warning）
- [ ] 测试使用真实代码（mock 仅限不可避免的场景）
- [ ] 边界场景和错误路径已覆盖
- [ ] 测试命名遵循 `should_X_when_Y` 规范

无法全部勾选？你跳过了 TDD。重新开始。

## 最终规则

```
生产代码 → 测试存在且先失败过
否则 → 不是 TDD
```

没有豁免（除非用户明确批准）。
