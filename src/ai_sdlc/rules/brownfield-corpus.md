# 存量工程认知基线（Brownfield Corpus）

> 在**已有业务代码**的仓库上执行标准 SDLC 之前，必须先产出 `.ai-sdlc/memory/engineering-corpus.md`，作为人与 Agent 的**共同索引**。禁止用「通读全仓库」代替该文档。

## 1. 何时强制生效

满足**全部**条件时为硬门禁（见 `quality-gate.md` INIT 阶段）：

1. 项目根目录或约定源码目录下**已存在业务代码**（Step 0 检查 G=是）；
2. 当前走**标准 SDLC**（非 `scenario-routing.md` §2 热修复短路径）。

以下情况**不要求**产出 corpus：

- **全新空项目**（G=否）；
- **热修复短路径**：以事件单与最小变更为准；事后可 backlog 补 corpus；
- `engineering-corpus.md` 已存在且通过新鲜度检查（见 §4）。

## 2. 单一事实来源

- **路径**：`.ai-sdlc/memory/engineering-corpus.md`
- **模板**：`templates/brownfield-corpus-template.md`
- **与宪章的关系**：宪章回答「必须遵守什么」（规范性）；corpus 回答「现有系统是什么」（描述性）。Stage 0 生成宪章时须对照 corpus 中的事实，不得产生矛盾。
- **与 README 的关系**：独立成文。README 面向人类 onboarding；corpus 面向结构化推理与变更定位。二者可互相链接路径，不引用对方内容。若新建或更新项目根 `README.md`，**正文须使用简体中文**（见 `rules/pipeline.md`「产出语言」）；可参考 `templates/readme-template.zh.md`。

## 3. 分层有界探索

### 3.1 深度与变更范围挂钩

| 变更范围 | 探索深度 | 文件读取预算（参考值） |
|---------|---------|---------------------|
| 小范围修复 / 单模块改动 | Tier 1 | ≤ 10 文件 |
| 跨模块功能 / 中等重构 | Tier 2 | ≤ 30 文件 |
| 架构级重构 / 大规模新增 | Tier 3 | ≤ 50 文件 + 按需追加 |

### 3.2 各 Tier 覆盖范围

**Tier 1（结构层）**：目录树 + 依赖清单 + 已有文档 + 顶层 schema → 产出 §1-§2, §5 概要, §9, §10

**Tier 2（架构层）**：Tier 1 + 入口文件 + 路由/DI 配置 + 模块接口 + 配置文件 + ORM model/migration + CI 配置 → 产出 §3-§8, 附录 A

**Tier 3（细节层）**：Tier 2 + 变更涉及区域的核心业务逻辑、复杂算法、集成实现、测试套件 → 全章节完善, 附录 B

### 3.3 探索铁律

1. **每层有文件读取预算**。超出预算的区域写入 §10 Open Questions，不假装已读。
2. **只持久化不可推导的知识**。依赖列表、API 清单、文件树等可从源码实时获取的信息不写入 corpus。
3. **所有断言指向具体路径**。写「认证模块在 `src/auth/`」而非「系统有认证功能」。
4. **先索引后深入**。先建目录/模块级地图，再按需展开关键文件。
5. **显式标记未知**。无法确认的信息标注 UNKNOWN，不臆测。

### 3.4 必填章节与深度的关系

| 章节 | Tier 1 | Tier 2 | Tier 3 |
|------|:------:|:------:|:------:|
| §1-§5, §9, §10 | 必填 | 必填 | 必填 |
| §6-§8 | 可选（写 UNKNOWN） | 必填 | 必填 |
| 附录 A | 可选 | 建议 | 必填 |
| 附录 B | 可选 | 可选 | 建议 |

## 4. 维护策略

以规则文本约束为主体，辅以一项轻量硬检查。

### 4.1 规则约束（Agent 自觉执行）

以下情况应更新 corpus：

- 大范围重构完成后（新增/删除子系统、架构模式变更）
- 外部依赖发生框架级变更（如 Vue 2→3、Express→Fastify）
- 人工显式要求刷新时

小范围代码修改（bug fix、单文件改动、新增测试）不要求同步更新 corpus。

### 4.2 Stage 6 CLOSE 增量刷新（流水线内自动执行）

每次开发任务完成进入 Stage 6 CLOSE 时，若 `engineering-corpus.md` 存在，自动执行增量刷新：

1. 对照本次 feature 分支的文件变更（`git diff main...HEAD --stat`）与 corpus §2 仓库地图
2. 判断是否有结构性变更：新增/删除一级或二级目录、新增模块、入口文件变化、外部依赖框架级变更
3. **有结构性变更** → 更新 §2 仓库地图、§3 模块边界、§4 入口索引、附录 A 关键文件索引中受影响的条目；其他章节如明显过时也一并修正
4. **无结构性变更** → 仅更新 corpus 头部「最后验证」时间戳，表示 corpus 在此版本确认有效
5. 更新后的 corpus 随 CLOSE 的最终 `git commit` 一起提交

**注意**：这是增量更新（仅改受影响章节），不是重走 Tier 1/2/3 全量探索。目标是让 corpus 始终反映最新项目状态，使下一次开发任务进入时 Step 0 的新鲜度检查基本不会触发 STALE。

### 4.3 Step 0 硬检查（自动执行）

每次进入 Step 0 环境探测时，若 `engineering-corpus.md` 已存在，执行**一项**新鲜度验证：

```
扫描项目根目录的一级子目录列表（排除 .git、node_modules 等通用忽略目录），
与 corpus §2 仓库地图中列出的目录对比：
  - 若存在仓库地图中未出现的新一级目录 → 标记 STALE
  - 若仓库地图中的一级目录在实际文件系统中已不存在 → 标记 STALE
  - 否则 → 视为有效
```

**STALE 处理**：按当前 PRD 的变更范围决定刷新深度（Tier 1/2/3），刷新 corpus 后继续。

不引入 commit 计数、天数阈值等容易误报的机制。

## 5. 与后续阶段的关系

| 阶段 | 如何使用 corpus |
|------|----------------|
| Step 0.4 宪章生成 | 技术硬约束不得与 corpus 中已证实的仓库事实矛盾 |
| Stage 1 REFINE | 用户故事与验收不得与 corpus 已述边界矛盾；若需突破，在 spec 中显式记录变更理由 |
| Stage 2 DESIGN | plan/research 须引用 corpus 中的模块边界与接口 |
| Stage 3+ | 任务文件路径应优先落在 corpus 已映射区域；新区域先更新 corpus 再分配任务 |
| Stage 6 CLOSE | 增量刷新 corpus 受影响章节（§4.2）；无结构性变更则仅更新时间戳；随最终 commit 提交 |
