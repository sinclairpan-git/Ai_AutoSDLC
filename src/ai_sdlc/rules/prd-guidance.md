# PRD 引导协议

> 本规则在流水线启动前激活。当用户提供的输入不是完整 PRD 时，引导用户完成从想法到可执行 PRD 的全过程。当用户提供完整 PRD 时，审核并协助完善。

## 产出语言

最终 PRD 及由此衍生的说明性 Markdown **须以简体中文撰写**（产品名、专有名词、API 路径等可保留原文）；与 `rules/pipeline.md`「产出语言（默认）」一致。

## 核心理念

真正的自动化不从代码开始，从需求开始。

用户的输入千差万别：可能是一句话的灵感，可能是一段模糊的描述，也可能是一份专业产品经理写的完整 PRD。我们的工作是：无论输入质量如何，都确保进入流水线的 PRD 是完整、具体、可执行的。

**绝不在需求模糊的情况下启动流水线。** 模糊的需求只会产生模糊的代码。

## 触发条件

当用户向流水线提供输入时，首先进行输入分类：

```
输入分类：
  A. 一句话 / 模糊想法（如"做一个报名系统"、"我要一个 TODO 应用"）
     → 进入「探索模式」

  B. 粗略描述（有核心功能但缺少细节，如几段话描述功能）
     → 进入「结构化模式」

  C. 完整 PRD（已有用户角色、业务规则、验收标准等）
     → 进入「审核模式」

分类依据（自动判断）：
  - 输入 < 100 字且无结构化章节 → A
  - 输入 100-500 字或有部分结构 → B
  - 输入 > 500 字且包含至少 3 个必填章节 → C
  - 输入是文件路径且文件内容匹配 PRD 模板 → C
```

## 模式 A：探索模式（从一句话到完整 PRD）

### 阶段 1：理解核心意图

```
目标：理清用户到底想做什么

一次只问一个问题。优先选择题，开放题为辅。
每轮对话结束后，用一句话总结理解到的信息。

引导问题库（按需选用，不全部问）：

Q1：产品定位
  "这个产品面向谁用？"
  选项示例：
    a. 公司内部员工
    b. 外部客户 / 终端用户
    c. 开发者 / 技术人员
    d. 其他（请说明）

Q2：核心问题
  "它主要解决什么问题？如果只能做一件事，那件事是什么？"

Q3：产品形态
  "你设想的是什么形态？"
  选项：
    a. Web 应用（浏览器访问）
    b. 移动端应用 / H5
    c. 命令行工具
    d. API 服务 / 后端
    e. 桌面应用
    f. 其他

Q4：用户规模
  "大概有多少用户会使用？"
  选项：
    a. < 50 人（小团队）
    b. 50-500 人（中型组织）
    c. 500+ 人（大型组织）
    d. 不确定

Q5：已有约束
  "有任何技术约束或偏好吗？比如必须用某个框架、要对接某个系统、或者要在某个平台上运行？"
```

### 阶段 2：展开功能

```
目标：把核心意图展开为具体功能

基于阶段 1 的信息，提出 2-3 个可能的功能范围方案：

方案模板：
  方案 A（最小 MVP）：
    - 功能 1：[描述]
    - 功能 2：[描述]
    - 不做：[明确排除项]
    - 预估复杂度：[低/中/高]

  方案 B（标准版本）：
    - 功能 1-3：[同 A]
    - 功能 4：[额外功能]
    - 功能 5：[额外功能]
    - 不做：[明确排除项]
    - 预估复杂度：[中/高]

  方案 C（完整版）：（仅在用户需求明显复杂时提供）
    ...

提问：
  "以上哪个方案更接近你的想法？或者有什么需要调整的？"

推荐：
  明确推荐一个方案并说明理由（通常推荐 A 或 B，不推荐最复杂的）
```

### 阶段 3：细化关键维度

```
基于用户选定的方案，逐维度细化（每次一个维度）：

维度清单：
  1. 用户角色和权限
     "这个系统有哪些不同的角色？每个角色分别能做什么？"

  2. 核心业务规则
     "有什么约束和规则？比如：报名有没有截止时间？一个人能报多少次？"

  3. 页面/交互
     "用户打开应用后，看到的第一个页面是什么？然后怎么操作？"

  4. 数据流向
     "数据从哪来？到哪去？需要和其他系统对接吗？"

  5. 异常处理
     "如果 [核心操作] 失败了，应该怎么处理？用户看到什么？"

  6. 验收标准
     "怎么判断这个功能'做好了'？你最在意哪几个点？"

每个维度确认后再进入下一个。
```

### 阶段 4：生成 PRD 并确认

```
基于以上对话，按 templates/prd-template.md 格式生成完整 PRD。

生成后展示摘要：
  "基于我们的讨论，我生成了完整的 PRD。核心要点如下：
    - 产品：[名称]，[形态]
    - 目标用户：[角色 1, 角色 2]
    - 核心功能：[功能 1, 功能 2, ...]
    - 本期不做：[排除项 1, 排除项 2, ...]
    - 验收标准：[N 条]

  你想：
    [查看] → 查看完整 PRD 内容
    [调整] → 我来修改某些部分
    [启动] → PRD 已完善，启动自动化流水线"

只有用户明确选择 [启动] 后，才进入流水线 Stage 0。
```

## 模式 B：结构化模式（从粗略描述到完整 PRD）

```
用户已有部分描述但不完整。

Step 1：解析现有内容
  - 识别已包含的 PRD 维度
  - 识别缺失的必填维度
  - 识别模糊需要澄清的部分

Step 2：输出缺口分析
  "你的描述已经覆盖了以下部分：
    ✅ 项目背景
    ✅ 核心功能
    ⚠️ 用户角色（提到了但不完整）
    ❌ 业务规则（缺失）
    ❌ 验收标准（缺失）
    ❌ 本期不做（缺失）

  我们需要补充 [N] 个关键部分。可以逐个来吗？"

Step 3：逐个补充缺失维度
  - 每次一个维度
  - 提供选择题 + 建议
  - 用户确认后进入下一个

Step 4：生成完整 PRD 并确认（同模式 A 阶段 4）
```

## 模式 C：审核模式（完整 PRD 的质量审核）

```
用户提供了完整 PRD。不直接启动流水线，先审核。

Step 1：质量评分
  按以下维度评分（每项 1-5 分）：

  | 维度 | 评分标准 |
  |------|---------|
  | 完整性 | 必填章节是否齐全 |
  | 具体性 | 业务规则是否具体到可测试 |
  | 明确性 | 是否有歧义或自相矛盾 |
  | 可测试性 | 验收标准是否可量化 |
  | 范围控制 | "本期不做"是否明确 |
  | 一致性 | 各章节之间是否矛盾 |

  总分 25-30 分 → 优秀
  总分 18-24 分 → 良好，有改进空间
  总分 < 18 分 → 需要补充

Step 2：输出审核结果
  "PRD 审核完成。总分：[X]/30

  ✅ 亮点：
    - [做得好的部分]

  ⚠️ 建议改进（[N] 项）：
    1. [建议 1]：[具体说明]
    2. [建议 2]：[具体说明]

  ❌ 必须补充（[N] 项）：
    1. [缺失 1]：[为什么重要]

  你想：
    [完善] → 我们一起补充和改进上述问题
    [启动] → PRD 已经够好了，直接启动流水线
    [查看详情] → 查看每项评分的详细说明"

Step 3：如果用户选择 [完善]
  - 逐项引导补充
  - 每项补充后更新 PRD
  - 全部补充完毕后重新展示确认

Step 4：如果用户选择 [启动]
  - 即使有改进建议，尊重用户决定
  - 将未改进的部分作为 WARNING 记录，供 Stage 1 REFINE 参考
  - 进入流水线 Stage 0
```

## 对话风格指引

```
原则：
  1. 一次只问一个问题 — 不要用一堆问题轰炸用户
  2. 优先选择题 — 降低用户思考负担
  3. 给推荐意见 — 不是所有选项平等对待，要有倾向
  4. 用用户的语言 — 不用技术术语，除非用户先用了
  5. 总结确认 — 每轮对话结束用一句话确认理解
  6. 不假设 — 不确定就问，不替用户决定

反模式：
  ❌ 一次问 5 个问题
  ❌ 所有选项一样权重不给建议
  ❌ 用技术术语回应非技术用户
  ❌ 没确认就假设用户意图
  ❌ 在用户说 "可以了" 前自行启动流水线
```

## 输出标准

无论通过哪种模式，最终输出的 PRD 必须满足：

```
最低质量门禁（进入 Stage 0 的前提）：

必须包含：
  □ 项目名称和产品形态
  □ 至少 1 个明确的产品目标
  □ 至少 1 个用户角色（含操作列表）
  □ 至少 3 条业务规则（具体到可测试）
  □ "本期不做"清单（至少 1 项）
  □ 至少 3 条验收标准（可量化）
  □ 开发优先级（P0 / P1 分级）

建议包含（缺失不阻断，但记录 WARNING）：
  □ 页面/交互描述
  □ 非功能需求
  □ 外部集成
  □ 技术偏好
```

## 特殊场景处理

### 用户说"你自己决定"

```
当用户不愿回答细节问题时（如"你看着办"、"你决定"）：
  1. 不阻断
  2. AI 基于行业常见实践做出合理假设
  3. 将每个假设显式记录为 AI 决策
  4. 在 PRD 对应位置标注：[AI 推断：基于 XX 假设]
  5. 生成后让用户确认
```

### 用户着急想跳过

```
当用户表示"别问了直接开始"：
  1. 评估当前 PRD 质量是否达到最低门禁
  2. 如果达到 → 尊重用户意愿，启动流水线
  3. 如果未达到 → 说明缺少什么以及为什么重要，给出两个选项：
     a. 花 2-3 分钟补充关键信息（告知具体缺什么）
     b. AI 基于假设填充，但标注为推断（风险更高）
```

### 已有代码想加新功能

```
当用户在已有项目中提出新功能需求时：
  1. 不走完整 PRD 引导（项目初始化已完成）
  2. 只关注新功能的增量需求
  3. 简化流程：理解功能 → 确认范围 → 确认约束 → 生成增量 PRD
  4. 增量 PRD 只需包含新功能相关的章节
```

## 与流水线的集成

```
PRD 引导 → 输出完整 PRD 文件
  ↓
用户确认 [启动]
  ↓
Stage 0 INIT 读取 PRD → 开始自动化流水线
```

PRD 引导是流水线的「前置阶段」，不编号（不是 Stage -1，而是流水线的入口门卫）。

在 `autopilot.md` 的 Step 0 之前执行。
