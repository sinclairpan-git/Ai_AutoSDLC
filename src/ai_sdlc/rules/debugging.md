# 系统化调试协议

> 本规则在 EXECUTE 阶段的调试循环中激活。当测试失败、构建报错或出现非预期行为时，必须遵循本协议。
> **违反本规则的字面要求即违反其精神。**

## 铁律

```
没有根因调查，就不能提出修复方案。
```

如果没有完成阶段一，不能动手改代码。

## 核心原则

随机修复浪费时间并引入新 bug。快速补丁掩盖底层问题。

**ALWAYS 找根因 BEFORE 尝试修复。** 修复症状即失败。

## 何时使用

**所有技术问题：**
- 测试失败
- 构建失败
- 非预期行为
- 性能问题
- 集成问题

**尤其是当：**
- 时间紧迫（越急越容易猜测）
- "显而易见"只需改一行（显而易见 = 危险信号）
- 已经尝试了多次修复
- 上一次修复没效果
- 你不完全理解问题

**不要跳过：**
- "问题很简单" — 简单问题也有根因
- "太赶了" — 系统化调试比乱猜更快
- "先试一下再调查" — 第一次修复的方式决定了后续所有修复的方式

## 四阶段流程

必须按顺序完成每个阶段，不可跳跃。

### 阶段一：根因调查

**在尝试任何修复之前：**

**1. 仔细阅读错误信息**
```
不要跳过错误或警告。错误信息常常直接包含答案。
完整阅读堆栈跟踪。
记录行号、文件路径、错误码。
```

**2. 稳定复现**
```
能可靠地触发吗？
准确的复现步骤是什么？
每次都发生吗？
如果不能复现 → 收集更多数据，不要猜。
```

**3. 检查近期变更**
```
什么变了可能导致这个问题？
git diff，最近的 commit
新的依赖、配置变更
环境差异
```

**4. 多组件系统中的证据收集**

**当系统有多个组件（API → 服务 → 数据库）：**

在提出修复之前，添加诊断埋点：

```
对每个组件边界：
  - 记录进入组件的数据
  - 记录离开组件的数据
  - 验证环境/配置传播
  - 检查每层的状态

运行一次收集证据 → 显示 WHERE 断裂
然后分析证据 → 定位失败组件
然后调查该组件
```

**5. 向上追溯数据流（根因追溯）**

Bug 常常在调用链深处表现出来。不要在症状点修复。

```
追溯协议：
  1. 观察症状 — 错误是什么？
  2. 找到直接原因 — 什么代码直接导致了错误？
  3. 问：谁调用了它？传了什么值？
  4. 继续向上追溯 — 直到找到错误数据的源头
  5. 在源头修复，不在症状点修复

示例：
  症状：git init 在源码目录执行（而非临时目录）
  追溯：
    git init cwd=projectDir → projectDir = ''
    → Session.create() 传了空字符串
    → context.tempDir 在 beforeEach 之前被访问
    → setupCoreTest() 初始返回 { tempDir: '' }
  根因：顶层变量在初始化前被访问
  修复：改为 getter，在 beforeEach 前访问直接抛错
```

### 阶段二：模式分析

**在修复之前找到模式：**

**1. 找到有效的类似代码**
```
同一代码库中有类似的正常工作的代码吗？
什么正常 vs 什么坏了？
```

**2. 对比参考实现**
```
如果在实现某个模式 → 完整阅读参考实现
不要略读 — 逐行阅读
在应用之前充分理解模式
```

**3. 识别差异**
```
正常代码和问题代码之间有什么不同？
列出每个差异，无论多小
不要假设"那个不可能有影响"
```

**4. 理解依赖**
```
这个组件需要什么其他组件？
需要什么配置、环境变量？
做了什么隐式假设？
```

### 阶段三：假设与验证

**科学方法：**

**1. 形成单一假设**
```
清晰陈述："我认为 X 是根因，因为 Y"
写下来（记录在 execution-log 中）
要具体，不含糊
```

**2. 最小化测试**
```
做最小可能的变更来验证假设
一次只改一个变量
不要同时修多个东西
```

**3. 验证后再继续**
```
生效了？→ 阶段四
没生效？→ 形成新假设
不要在无效修复上继续叠加
```

**4. 不知道时就说不知道**
```
说"我不理解 X"
不要假装知道
查阅文档、扩大调查范围
```

### 阶段四：实现修复

**修根因，不修症状：**

**1. 创建失败测试用例**
```
最简复现（遵循 TDD 规则）
自动化测试（如有测试框架）
必须在修复前存在
→ 引用 rules/tdd.md 的 Red-Green-Verify 循环
```

**2. 实施单一修复**
```
解决已确认的根因
一次一个变更
不做"顺便"的改进
不捆绑重构
```

**3. 验证修复**
```
测试通过了？
其他测试没坏？
问题真的解决了？
```

**4. 修复无效时**
```
停下来。
计数：你已经尝试了几次修复？
< 3 次：回到阶段一，用新信息重新分析
≥ 3 次：停下来，质疑架构（步骤 5）
不要在没有架构讨论的情况下尝试第 4 次修复
```

**5. 3 次以上修复失败：质疑架构**

```
指示架构问题的信号：
  - 每次修复都暴露新的共享状态/耦合问题
  - 修复需要"大规模重构"才能实现
  - 每次修复在其他地方产生新症状

停下来质疑根本面：
  - 这个模式是否根本站不住脚？
  - 是否因为惯性在坚持错误方向？
  - 应该重构架构 vs 继续修补症状？

→ 阻断流水线，输出诊断，要求用户介入
```

### 修复后：防御纵深

修复根因后，在调用链的每一层添加防御：

```
示例（项目目录传空值的 bug）：
  修复根因后，额外添加：
  Layer 1: Project.create() 验证目录非空
  Layer 2: WorkspaceManager 验证路径存在
  Layer 3: NODE_ENV 守卫拒绝在非临时目录执行危险操作
  Layer 4: 危险操作前的堆栈跟踪日志

原则：即使根因不再触发，防御层确保同类 bug 不可能再出现
```

## 与批次协议的集成

在 `batch-protocol.md` 的调试循环中，本协议替代原有的简单调试流程：

```
Round 1：
  - 执行阶段一（根因调查）
  - 执行阶段二（模式分析）
  - 执行阶段三（假设验证）
  - 执行阶段四（实现修复）
  - 运行测试

Round 2（如仍失败）：
  - 回到阶段一，扩大调查范围
  - 新假设 + 新修复
  - 运行测试

Round 3（如仍失败）：
  - 评估是否为架构问题（阶段四 步骤 5）
  - 如果是 → 阻断流水线
  - 如果不是 → 最后一轮修复

Round 3 后仍失败 → 触发熔断器，阻断流水线
```

## 调试日志记录格式

在 execution-log 中记录每次调试：

```markdown
#### 调试记录 | T0XX [任务名]

**症状**：[错误信息/现象]
**根因调查**：
  - 错误信息分析：[关键信息]
  - 复现：[是/否]，[步骤]
  - 近期变更：[git diff 摘要]
  - 数据流追溯：[A → B → C → 根因位置]
**假设**：[X 是根因，因为 Y]
**验证**：[最小变更描述] → [结果]
**修复**：[实际修改]
**防御层**：[添加的额外防御]
**测试确认**：V1 通过 / V2 通过
```

## 红旗 — 立即停下并回到阶段一

如果你在想：
- "先试改一下看看"
- "快速修复，之后再调查"
- "同时改多个地方，跑下测试"
- "跳过测试，手动验证就好"
- "大概是 X，先改了再说"
- "我不完全理解但这也许有效"
- "文档说 X 但我换个方式"
- "以下是主要问题：[直接列修复方案而没有调查]"
- 在追溯数据流之前就提出解决方案
- **"再试一次修复"（已经尝试了 2 次以上）**
- **每次修复都暴露不同位置的新问题**

**以上全部意味着：停下。回到阶段一。**

**如果已尝试 3 次以上修复：质疑架构。**

## 自我合理化防御表

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要走流程" | 简单问题也有根因。流程对简单 bug 很快。 |
| "紧急，没时间走流程" | 系统化调试比猜-试-猜循环更快。 |
| "先试这个再调查" | 第一次修复的方式决定模式。一开始就做对。 |
| "修完再补测试" | 没测试的修复不可靠。先写测试。 |
| "同时改多个省时间" | 无法隔离哪个生效。会引入新 bug。 |
| "参考文档太长了，我改编一下" | 部分理解保证出 bug。完整阅读。 |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（第 3 次后） | 3+ 次失败 = 架构问题。质疑模式，别再修了。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 读错误、复现、查变更、收集证据、追溯数据流 | 理解 WHAT 和 WHY |
| **2. 模式** | 找有效代码、对比差异 | 识别差异 |
| **3. 假设** | 形成理论、最小验证 | 已确认或形成新假设 |
| **4. 实现** | 写测试、修复、验证、防御纵深 | bug 消除，测试通过 |
